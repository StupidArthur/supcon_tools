## 一、问题一句话概括

**精馏塔模型在"直接流量模式"下能稳定运行，但切换到"阀位模式"后立即崩溃**：
压力从 101 kPa 急跌到 70 kPa（下限），回流罐液位从 50% 跌到 16%，塔釜液位从 50% 涨到 87%，
整个过程在 1000~2000 周期（500~1000 s）内发生。

切换前后的同一塔、同一状态、同一目标流量，仅仅因为"流量来源"从外部直接输入变成"由阀位反算"，结果天差地别。

## 二、模型基本信息

### 2.1 设备规模

- 乙醇—水物系，12 块理论板 + 进料板第 7 块（0-indexed 第 6 块）
- 全凝器 + 回流罐 + 釜式再沸器
- 塔顶压力 101.325 kPa(a)
- 进料 100 kg/h，乙醇质量分数 0.25，进料温度 60 ℃（过冷）
- 设计目标：xD=0.85, xB=0.015, R=3.0

### 2.2 模型结构

| 模块 | 实现 |
|---|---|
| 热力学 | Antoine + NRTL（已验证，23 点 VLE MAE=0.21 K） |
| 塔板动态 | 逐板物料 + 能量衡算（Heun 法子步积分） |
| 气相库存 | N_vapor、nE_vapor、U_vapor 三状态（理想气体状态方程） |
| 再沸器 | Q_R_available = (ṁ_steam/3600) × ΔH_steam × η |
| V_boil | V_boil = Q_for_vaporization / ΔH_vap_sump（含 Q_loss、Q_subcool） |
| 冷凝器 | Q_C = min(Q_flow, Q_UA)，V_condense = min(capacity, vapor_available) |
| 阀门 | 归一化等百分比 f(x) = (R^x-1)/(R-1)，一阶响应 τ=满行程/5 |

### 2.3 两种执行模式

```python
# 模式 A：直接流量模式（向后兼容测试用）
col.execute(
    feed_flow_kgmol_per_s=F,
    reflux_flow_kgmol_per_s=R,
    distillate_flow_kgmol_per_s=D,
    bottoms_flow_kgmol_per_s=B,
    vapor_boilup_kgmol_per_s=V,          # ← V 直接注入
)

# 模式 B：阀位模式（正式 DSL 用）
col.execute(
    feed_valve_pct=...,
    reflux_valve_pct=...,
    distillate_valve_pct=...,
    bottoms_valve_pct=...,
    steam_valve_pct=...,                  # ← V 由 Q_R 反算
    cooling_valve_pct=...,
)
```

**关键代码差异（model 内部 `execute()` 方法）：**

```python
if any_valve_input:
    # 阀位模式：V_boil 由 Q_R 真实机理计算
    direct_vapor_bypass = None
    self._last_steam_flow_kg_per_h = (
        self._valve_max_flow_kg_per_h["steam"] * self._valves["steam"].get_flow_fraction()
    )
else:
    # 直接流量模式：V_boil 由外部注入（bypass Q_R 机理）
    if vapor_boilup_kgmol_per_s is not None:
        direct_vapor_bypass = self._last_vapor_boilup
    else:
        direct_vapor_bypass = None
```

## 四、核心诊断：为什么会崩溃？

### 4.1 直接观测到的现象

1. **第一周期压力就掉了 3.5 kPa**（101.32 → 97.78），说明 V_boil 在阀位模式下立即低于直接流量模式
2. **200 周期内压力跌到 70 kPa 下限**，说明 V_boil 持续偏低或 V_condense 持续偏高
3. **回流罐抽空 + 塔釜满液**，说明物料平衡被打破：D 流不出去，B 流不动，进料持续进入

### 4.2 我怀疑的根因（但未完全证实）

#### 怀疑 1：阀门一阶响应导致 actual_pct 实际偏低

虽然我设置了 `valve.command_pct = 85.85` 和 `valve.actual_pct = 85.85`，
但在 `execute()` 内部又会调用 `valve.update(cycle_time)`，这是正确的一阶响应。

理论上如果 `command == actual`，`update()` 不应该改变 `actual`，所以这点应该没问题。

#### 怀疑 2：Q_R → V_boil 反算路径有 BUG

直接流量模式下：
```python
V_boil = direct_vapor_bypass = 0.000837 kmol/s  # 直接注入
```

阀位模式下：
```python
steam_flow_kg_h = 100.0 * valve.get_flow_fraction()  # valve at 85.85% → ~60.49 kg/h
Q_R_available = (60.49 / 3600) * 2133.0 * 0.95 = 34.0 kW
Q_loss_sump = UA_sump * (T_sump - T_amb) ≈ 0.02 * (373 - 298) = 1.5 kW
Q_subcool ≈ 0  # 塔釜接近泡点
Q_for_vaporization = 34.0 - 1.5 = 32.5 kW
V_boil = 32.5 / ΔH_vap_sump ≈ 32.5 / 38800 = 0.000838 kmol/s  # 应该接近 0.000837
```

**理论上 V_boil 应该几乎一致**，但实际表现却差很多。怀疑：
- 反算 steam_flow_kg_h 时用了当前 sump 组成对应的 ΔH_vap，而阶段 1 末态的 sump 组成可能不是设计点
- 或者 Q_subcool 计算有误，在阶段 1 末态 sump 温度低于泡点，导致 Q_subcool 占用大量 Q_R

#### 怀疑 3：阶段 1 末态根本不是稳态

阶段 1 的指标看起来"稳定"，但可能只是慢动态。xB=0（过分离），xD 从 0.89 → 0.73（持续下降），
说明塔板组成剖面仍在缓慢演化。切换到阀位模式后，组成继续变化导致 V_boil 估算偏离。

#### 怀疑 4：阀位模式下过程阀流量组成换算用了"上一周期"的 top_wt/bottom_wt

```python
top_wt = float(self.top_ethanol_wt) if hasattr(self, "top_ethanol_wt") else 0.85
bottom_wt = float(self.bottom_ethanol_wt) if hasattr(self, "bottom_ethanol_wt") else 0.015
reflux_mw = _mixture_molecular_weight(ethanol_mass_fraction_to_mole_fraction(top_wt))
distillate_mw = reflux_mw
bottoms_mw = _mixture_molecular_weight(ethanol_mass_fraction_to_mole_fraction(bottom_wt))
```

这意味着 reflux/distillate/bottoms 的摩尔流量换算用了上一周期的组成，
但阶段 1 的 xD=0.73，与设计 xD=0.85 偏差大，导致同样的阀位下摩尔流量与直接流量模式不一致。

#### 怀疑 5：根本是策略问题——阶段 1 用直接流量模式本身就是错的

直接流量模式绕过了 Q_R 机理，相当于"虚拟"地把 V_boil 钉死在 0.000837。
此时蒸汽阀、塔釜能量动态、气相动态都是"被动跟随"。
切换到阀位模式后，所有这些机理同时激活，任何一处不匹配都会导致失稳。

**正确做法可能是**：从一开始就在阀位模式下跑，用 PI 控制器调节阀位而不是流量。
但这又有鸡生蛋问题：没有稳态怎么反算初始阀位？

## 五、已经尝试过的策略（都失败）

1. **多回路 PI（5 回路：P→冷却, drum→D, sump→B, xD→R, xB→V）**：强非线性导致振荡
2. **迭代比例修正 R/V**：V < R 导致回流罐抽空
3. **二分搜索 R**：固定 D/B 时过冷进料导致塔釜液位漂移；xD 非单调
4. **4 回路 PI + CMO 代数约束 V = R + D**：正反馈循环，R 和 D 均被驱至下限
5. **4 回路 PI + 固定 V（直接流量模式 → 阀位模式切换）**：当前问题，模式切换崩溃

## 六、想请教的问题

### Q1：模式切换崩溃的根因到底是什么？

理论上"由 V 反推 steam_flow_kg_h"再切换到阀位模式，V_boil 应该几乎不变。
但实际崩溃如此之快（200 周期内压力掉到下限），说明 V_boil 偏差很大。

请帮忙分析：在 VLE/能量衡算/阀门动态层面，有没有我遗漏的耦合？

### Q2：稳态生成的标准做法是什么？

我现在的策略是"先用直接流量模式松弛，再反算阀位切换"。
但模式切换的瞬间崩溃让我怀疑这个流程本身就有问题。

**正确的工业实践是**：
- (a) 一开始就在阀位模式下松弛（用 PI 调节阀位），等阀门稳定后保存状态？
- (b) 用代数稳态求解器（Newton-Raphson）直接解稳态方程，不走动态松弛？
- (c) 还是有其他更稳妥的"两阶段"切换方法？

### Q3：12 块理论板 + R=2~3 的过分离问题怎么处理？

设计目标是 xD=0.85, xB=0.015。
但实际跑出来 xB 总是趋近 0（甚至 < 1e-5），xD 总是 0.85~0.89。
说明塔的分离能力远超设计目标，xB 由 VLE 决定而不是塔板数决定。

是否应该：
- (a) 放宽 xB 收敛阈值到 0~0.02？
- (b) 调整进料组成或塔板数？
- (c) 接受 xB→0 的事实，只验收 xD 和物料平衡？

### Q4：xB 收敛阈值 < 0.020 是否合理？

todo/5.md §9.4 要求 0.010 ≤ xB ≤ 0.020。
但 VLE 决定的 xB ≈ 1e-5，根本进不了这个区间。
这是规格错误，还是需要调整 VLE 参数或操作点？

### Q5：直接流量模式 vs 阀位模式，V_boil 一致性怎么保证？

模型里 direct_vapor_bypass 路径直接把 V_boil 当外部输入，
而阀位模式 V_boil 由 Q_R_available 计算（含 Q_loss、Q_subcool 等扣减）。

即使我"反算 steam_flow_kg_h = V × ΔH_vap_sump × 3600 / (ΔH_steam × η)"，
也忽略了 Q_loss_sump 和 Q_subcool，导致切换后 V_boil 偏小。

**这种"两个模式并存"的设计是否合理？** 还是应该统一用一种模式？
