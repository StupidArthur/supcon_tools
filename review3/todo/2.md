## 2. 修复 AUTO 模式外部写 SV 无效

### 2.1 问题原因

当前 Engine 的外部写值通过以下方式修改实例：

```python
setattr(instance, attr_name, value)
```

因此通过 UA 写：

```text
pid1.SV = 1.5
```

只会修改：

```python
self.SV
```

新 PID 的 AUTO 模式却使用：

```python
self._local_sv
```

当前只有 `execute(SV=...)` 收到参数时才更新 `_local_sv`：

```python
if SV is not None:
    self._local_sv = float(SV)
```

随后 `_resolve_effective_sv()` 又使用旧 `_local_sv` 覆盖 `self.SV`，导致 UA 写值丢失。

### 2.2 设计约束

需要同时满足：

* AUTO 模式允许 UA 直接写 `pid.SV`。
* YAML 中配置了 `inputs.SV` 时，连接输入优先于 UA 属性写值。
* CAS/RCAS 模式下，对外 `SV` 仍镜像 `CSV`。
* CAS/RCAS 返回 AUTO 时，不能把镜像的 CSV 误保存成原本地 SV。
* CAS 模式下写 `SV` 可以更新“未来返回 AUTO 时使用的本地 SV”，但不能改变当前 CAS 有效设定值。

### 2.3 新增内部状态

在 `__init__()` 中增加：

```python
self._last_published_sv = float(self.SV)
```

该属性表示 PID 在上一周期主动发布到 `self.SV` 的值，用于识别周期之间是否发生外部写入。

不得加入 `stored_attributes`。

### 2.4 SV 输入捕获规则

在每周期 `_resolve_effective_sv()` 之前执行以下逻辑。

建议增加独立方法：

```python
def _capture_local_sv(
    self,
    sv_input: Optional[float],
) -> None:
    ...
```

处理顺序：

```python
if sv_input is not None and finite(sv_input):
    # YAML/DSL 显式输入优先
    self._local_sv = float(sv_input)

elif finite(self.SV) and self.SV != self._last_published_sv:
    # 没有连接输入，但 self.SV 与上周期发布值不同，
    # 说明可能由 UA/Engine 外部写入
    self._local_sv = float(self.SV)
```

不要在 `_apply_inputs()` 中直接丢失“SV 参数是否为 None”这一信息。

可以调整为：

```python
def _apply_inputs(...):
    ...
    if SV is not None and finite(SV):
        self._local_sv = float(SV)
```

并在调用 `_apply_inputs()` 前保存：

```python
externally_visible_sv_before_inputs = self.SV
```

或者将 SV 捕获完整放入单独方法。

### 2.5 发布值更新

每周期完成 `_resolve_effective_sv()` 后执行：

```python
self._last_published_sv = float(self.SV)
```

所有正常 return 分支都必须完成该更新，包括：

* OOS
* IMAN
* TR
* MAN
* AUTO
* CAS
* RCAS
* ROUT

建议不要在多个 return 前重复编写，可增加：

```python
def _publish_mode_state(self, mode: int) -> None:
    self.AUTO = mode
    self.CAS = 1 if mode == 6 else 0
    self._last_published_sv = float(self.SV)
```

或者在每个 return 前显式处理。

### 2.6 来源优先级

最终行为必须为：

| 情况                         | 本地 SV 来源                       |
| -------------------------- | ------------------------------ |
| `execute(SV=...)` 有值       | 本周期输入值                         |
| 没有 SV 输入，`self.SV` 被 UA 改写 | UA 写入值                         |
| 没有输入，也没有外部改写               | 原 `_local_sv`                  |
| MODE=6/7                   | 当前有效 SV 使用 CSV                 |
| MODE=6/7 时写 SV             | 保存到 `_local_sv`，当前有效 SV 仍为 CSV |

## 3. 修复首次 AUTO 启动没有比例响应

### 3.1 问题原因

当前代码将以下两种情况混为一体：

```python
not self._initialized
```

和：

```python
self._previous_mode not in _AUTOMATIC_MODES
```

首次直接以 AUTO 启动时，代码执行无扰初始化：

```python
self._prev_error_pct = current_error_pct
p_delta = 0.0
d_delta = 0.0
```

因此首次比例增量被抹掉。

如果：

```text
TI=0
TD=0
```

并且初始误差保持不变，那么之后：

```text
p_delta=0
i_delta=0
d_delta=0
```

控制器永远没有输出。

### 3.2 正确区分两种初始化

#### 情况 A：首次直接以自动类模式启动

包括：

```text
MODE=5
MODE=6
MODE=7
```

必须允许比例响应：

```python
p_delta = error_pct - 0.0
```

为避免冷启动微分冲击，本周期可以关闭微分：

```python
d_delta = 0.0
```

积分项按正常规则计算：

```python
i_delta = cycle_time / TI * error_pct
```

建议逻辑：

```python
if not self._initialized:
    self._mv_pct = self._engineering_mv_to_pct(self.MV)
    self._prev_error_pct = 0.0
    self._prev_derivative_state = 0.0

    p_delta = error_pct
    d_delta = 0.0
```

#### 情况 B：从非自动模式切换到自动类模式

例如：

```text
MAN -> AUTO
TR -> AUTO
IMAN -> CAS
ROUT -> AUTO
OOS -> AUTO
```

才执行无扰切换：

```python
self._prepare_bumpless_transfer(error_pct)
p_delta = 0.0
d_delta = 0.0
```

### 3.3 建议分支结构

修改为：

```python
if not self._initialized:
    # 冷启动自动：保留首次比例响应
    self._mv_pct = self._engineering_mv_to_pct(float(self.MV))
    self._prev_error_pct = 0.0
    self._prev_derivative_state = 0.0

    p_delta = error_pct
    d_delta = 0.0

elif self._previous_mode not in _AUTOMATIC_MODES:
    # 手动类/停止类进入自动：无扰
    self._prepare_bumpless_transfer(error_pct)

    p_delta = 0.0
    d_delta = 0.0

else:
    # 连续自动运行
    p_delta = error_pct - self._prev_error_pct
    d_delta = self._calculate_filtered_derivative(error_pct)
```

### 3.4 验收行为

以下控制器：

```text
MODE=5
PB=100
TI=0
TD=0
SWPN=1
PV=0
SV=50
MV=0
SV量程=0～100
MV量程=0～100
```

第一次执行后应得到：

```text
error_pct = 50
Kp = 1
MV = 50
```

后续误差不变时，MV 保持 50。

## 4. 严格校验 MODE

### 4.1 当前问题

当前使用：

```python
int(MODE)
```

因此：

```text
MODE=5.9
```

会被接受为 AUTO，但公开值仍可能是：

```text
MODE=5.9
AUTO=5
```

### 4.2 合法条件

MODE 必须同时满足：

1. 是有限数值。
2. 数值本身是整数。
3. 整数位于 1～8。

建议统一方法：

```python
def _parse_mode(value: Any) -> Optional[int]:
    try:
        number = float(value)
    except (TypeError, ValueError):
        return None

    if not math.isfinite(number):
        return None

    if not number.is_integer():
        return None

    mode = int(number)

    if mode not in _VALID_MODES:
        return None

    return mode
```

所有 MODE 入口都使用同一个方法：

* `_validate_init_params()`
* `_validate_or_restore_runtime_params()`
* `_apply_inputs(MODE=...)`

不要在三个位置分别实现不同规则。

### 4.3 规范化输出

构造或运行时接受合法 MODE 后，统一写回：

```python
self.MODE = mode
```

派生：

```python
self.AUTO = mode
```

必须始终保证：

```python
int(self.MODE) == self.MODE
self.AUTO == self.MODE
```

### 4.4 非法运行时写值

如果上一合法模式为：

```text
MODE=5
```

外部写：

```text
MODE=5.9
```

下一周期应恢复：

```text
MODE=5
AUTO=5
```

不得截断为 5 后视为合法写入，也不得保留 5.9。

## 5. 处理在线量程变化

## 5.1 当前问题

内部 `_mv_pct` 是依据旧 `MVSCL/MVSCH` 计算的。

运行时修改：

```text
MVSCL
MVSCH
```

后，代码直接用新量程把旧 `_mv_pct` 转回工程量，造成 MV 跳变。

### 5.2 推荐处理方式

允许在线修改量程，但修改时重新对齐内部百分比状态。

在 `_validate_or_restore_runtime_params()` 中，保存更新前的有效参数：

```python
old_mvscl = self._last_valid_params["MVSCL"]
old_mvsch = self._last_valid_params["MVSCH"]
old_svscl = self._last_valid_params["SVSCL"]
old_svsch = self._last_valid_params["SVSCH"]
```

如果新参数全部合法，检查量程是否变化。

### 5.3 MV 量程变化

如果：

```python
new_mvscl != old_mvscl or new_mvsch != old_mvsch
```

则根据当前工程值 MV 和新量程重新计算：

```python
current_mv = clamp(self.MV, self.MVL, self.MVH)
self._mv_pct = (
    100.0
    * (current_mv - new_mvscl)
    / (new_mvsch - new_mvscl)
)
```

目标是量程变化时保持工程量 MV 连续。

例如：

```text
修改前：量程 0～100，MV=50，内部50%
修改后：量程 0～200
```

修改后内部应改为：

```text
25%
```

这样工程输出仍为 50，而不是跳到 100。

### 5.4 SV/PV 量程变化

如果：

```python
new_svscl != old_svscl or new_svsch != old_svsch
```

则下一个自动周期必须重新对齐：

```python
self._prev_error_pct = current_error_pct
self._prev_derivative_state = 0.0
```

并在该周期：

```python
p_delta = 0.0
d_delta = 0.0
```

防止仅因量程定义改变而产生虚假的比例和微分跳变。

建议设置内部标志：

```python
self._sv_scale_changed = True
```

在 `execute()` 计算出新 `error_pct` 后处理：

```python
if self._sv_scale_changed:
    self._prev_error_pct = error_pct
    self._prev_derivative_state = 0.0
    p_delta = 0.0
    d_delta = 0.0
    self._sv_scale_changed = False
```

允许积分按一个周期正常作用。

### 5.5 限幅变化

如果新 `MVL/MVH` 不再包含当前 MV，允许 MV 立即被限制到新的合法范围，这不视为异常突跳。

## 6. 非法参数恢复日志

### 6.1 增加 logger

在 `pid.py` 中使用项目现有日志工具：

```python
from components.utils.logger import get_logger

logger = get_logger("pid")
```

### 6.2 恢复时输出原因

当前 `_restore_last_valid_params()` 不知道是哪个参数错误。

调整校验代码，使恢复方法接收原因：

```python
def _restore_last_valid_params(
    self,
    reason: str,
) -> None:
    logger.warning(
        "PID运行参数无效，已恢复上次有效参数: %s",
        reason,
    )

    for key, value in self._last_valid_params.items():
        setattr(self, key, value)
```

示例日志：

```text
PID运行参数无效，已恢复上次有效参数: PB必须大于0，实际值=-10
```

```text
PID运行参数无效，已恢复上次有效参数: MODE必须为1～8整数，实际值=5.9
```

```text
PID运行参数无效，已恢复上次有效参数: MVH不得大于MVSCH，MVH=120，MVSCH=100
```

不要只输出“参数错误”，必须包含具体参数名和值。

一次非法写值恢复后，下一周期参数已经合法，不应重复刷同一条日志。

## 7. 输入有限性保护

补充处理直接属性写入的非有限值。

### 7.1 PV

PV 可以超量程，但不能是：

```text
NaN
+Inf
-Inf
```

如果 `execute(PV=...)` 收到非有限值，继续沿用上一有效 PV。

如果 UA 直接把 `self.PV` 写成非有限值，且本周期没有连接 PV 输入，应恢复上一有效 PV。

建议内部增加：

```python
self._last_valid_pv = float(self.PV)
```

每周期：

```python
if finite(self.PV):
    self._last_valid_pv = float(self.PV)
else:
    self.PV = self._last_valid_pv
    logger.warning(...)
```

### 7.2 SV、CSV、MV

同样不得让 NaN/Inf 进入：

* 量程换算
* `_clamp()`
* `_mv_pct`
* 最终 MV

至少保证：

```python
math.isfinite(self.MV)
```

始终成立。

## 10. 最终验收标准

修改完成后必须满足：

* `engine.override_variable("pid1.SV", value)` 在 AUTO 模式真实生效。
* CAS/RCAS 仍以 CSV 为当前有效设定值。
* CAS 返回 AUTO 时恢复保存的本地 SV。
* 首次 AUTO 启动存在正常比例响应。
* MAN/IMAN/TR/ROUT/OOS 切换到 AUTO/CAS 时仍然无扰。
* `TI=0、TD=0` 时，纯比例控制器可以正常控制。
* MODE 只接受整数 1～8。
* `MODE` 和 `AUTO` 永远一致。
* 在线修改 MV 量程不会无故改变当前工程量 MV。
* 非法运行参数恢复时有明确日志。
* NaN/Inf 不得传播到最终 MV。
* 原有水箱配置仍能稳定到目标液位。
* 所有新增和现有 PID 测试通过。
