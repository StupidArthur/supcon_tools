# 前端架构设计（多引擎扩展版）

## 一、设计背景

随着后端架构的重大变化（数据总线模式、ConfigServer、多引擎支持），前端需要提供完整的 CRUD 能力，支持：

1. **基础设施层组态管理**：Engine 配置、Storage 配置、OPCUA 配置
2. **逻辑定义层组态管理**：DSL 配置文件（命名空间组态）
3. **Engine 生命周期管理**：创建、启动、停止、删除
4. **服务管理**：StorageService、OPCUA Server 的启动和停止
5. **组态发布**：发布活动、全局位号唯一性检查
6. **实时监控**：引擎拓扑图、服务状态、诊断信息

### 2.2 核心功能模块

1. **组态管理模块**：基础设施层和逻辑定义层的 CRUD
2. **引擎管理模块**：Engine 实例的创建、启动、停止、删除
3. **服务管理模块**：StorageService、OPCUA Server 的管理
4. **实时监控模块**：引擎拓扑图、服务状态、诊断信息
5. **历史数据模块**：历史数据查询（保持现有功能）
6. **数据模拟模块**：数据模拟预览（保持现有功能）

#### 3.1.2 功能点

**基础设施层配置**：
- Engine 配置列表展示（表格或卡片）
- 添加/编辑/删除 Engine 配置
- Engine 配置表单（engine_id, kind, mode, namespaces, clock_config, replay_source 等）
- Storage 配置表单
- OPCUA 配置表单
- 保存基础设施层配置到服务器

**逻辑定义层配置**：
- 命名空间列表展示
- 创建/编辑/删除命名空间配置
- DSL 编辑器（代码编辑器，支持 YAML 语法高亮）
- DSL 验证（语法检查、语义检查）
- DSL 预览（预览生成的实例和表达式）

**发布组态**：
- 发布前检查（配置完整性、位号唯一性）
- 全局位号唯一性检查结果展示
- 发布按钮（触发发布活动）
- 发布结果展示（成功/失败、引擎启动状态）

## 四、API 接口设计

### 4.1 基础设施层配置 API

```typescript
// 获取基础设施层配置
GET /api/v2/config/infrastructure
Response: {
  engines: EngineInstanceConfig[],
  storage: StorageConfig,
  opcua: OPCUAConfig
}

// 保存基础设施层配置
POST /api/v2/config/infrastructure
Body: {
  engines: EngineInstanceConfig[],
  storage: StorageConfig,
  opcua: OPCUAConfig
}

// 获取单个 Engine 配置
GET /api/v2/config/infrastructure/engines/{engine_id}

// 创建/更新 Engine 配置
PUT /api/v2/config/infrastructure/engines/{engine_id}
Body: EngineInstanceConfig

// 删除 Engine 配置
DELETE /api/v2/config/infrastructure/engines/{engine_id}
```

### 4.2 逻辑定义层配置 API

```typescript
// 获取所有命名空间列表
GET /api/v2/config/logic/namespaces
Response: string[]  // ["ns1", "ns2", ...]

// 获取指定命名空间配置
GET /api/v2/config/logic/namespaces/{namespace}
Response: {
  namespace: string,
  content: string,  // YAML 内容
  parsed: ProgramConfig  // 解析后的配置对象
}

// 创建/更新命名空间配置
PUT /api/v2/config/logic/namespaces/{namespace}
Body: {
  content: string  // YAML 内容
}

// 删除命名空间配置
DELETE /api/v2/config/logic/namespaces/{namespace}

// 验证命名空间配置
POST /api/v2/config/logic/namespaces/{namespace}/validate
Body: {
  content: string
}
Response: {
  valid: boolean,
  errors: string[]
}
```

### 4.3 组态发布 API

```typescript
// 检查全局位号唯一性
POST /api/v2/config/publish/validate
Response: {
  valid: boolean,
  engine_count: number,
  tag_count: number,
  conflicts: string[]  // 冲突的位号列表
}

// 发布组态
POST /api/v2/config/publish
Response: {
  status: "published" | "failed",
  engine_count: number,
  tag_count: number,
  registry: GlobalRegistry,
  engine_start_results: {
    [engine_id: string]: "started" | "failed" | "already_running"
  }
}
```

### 4.4 引擎管理 API

```typescript
// 获取所有引擎列表
GET /api/v2/engines
Response: EngineStatus[]

// 获取单个引擎状态
GET /api/v2/engines/{engine_id}
Response: EngineStatus

// 启动引擎
POST /api/v2/engines/{engine_id}/start

// 停止引擎
POST /api/v2/engines/{engine_id}/stop

// 删除引擎
DELETE /api/v2/engines/{engine_id}

// 获取引擎诊断信息
GET /api/v2/engines/{engine_id}/diagnostic
Response: DiagnosticInfo
```

## 七、实时更新机制

### 7.1 WebSocket 连接

使用 WebSocket 实现实时状态更新：

```typescript
// hooks/useWebSocket.ts
const useWebSocket = (url: string) => {
  const [data, setData] = useState(null)
  const wsRef = useRef<WebSocket | null>(null)
  
  useEffect(() => {
    wsRef.current = new WebSocket(url)
    wsRef.current.onmessage = (event) => {
      setData(JSON.parse(event.data))
    }
    return () => {
      wsRef.current?.close()
    }
  }, [url])
  
  return data
}

// 使用示例
const engineStatus = useWebSocket('ws://localhost:8000/ws/engines/status')
const busStatus = useWebSocket('ws://localhost:8000/ws/bus/status')
```

### 7.2 轮询机制（备选）

如果 WebSocket 不可用，使用轮询：

```typescript
// hooks/usePolling.ts
const usePolling = (fetchFn: () => Promise<any>, interval: number) => {
  const [data, setData] = useState(null)
  
  useEffect(() => {
    const fetchData = async () => {
      const result = await fetchFn()
      setData(result)
    }
    
    fetchData()
    const timer = setInterval(fetchData, interval)
    return () => clearInterval(timer)
  }, [fetchFn, interval])
  
  return data
}
```
