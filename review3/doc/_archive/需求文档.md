# Data Factory Next 需求文档 (V2.0 - 多引擎架构版)

## 1. 背景与愿景

随着工业仿真规模的扩大，系统需要从“单体仿真引擎”向“分布式仿真平台”演进。原本的 `data_next` V1 版本实现了统一的表达式执行逻辑，但难以支持跨进程扩展、异构数据源集成（如回放历史数据）以及高并发的数据消费。

**V2.0 目标**：构建一个多引擎并发、全局命名空间隔离、基于统一数据总线的工业仿真底座。

## 2. 核心架构需求

### 2.1 分布式生产者 (Multi-Engine)
- **模拟引擎 (Simulation Engine)**：基于 DSL 定义，执行有状态的周期性逻辑计算。
- **回放引擎 (Playback Engine)**：支持将 Excel/CSV 历史数据按设定频率“回放”至系统，模拟真实传感器波形。
- **并发性**：支持在同一物理机或多台机器上启动任意数量的计算引擎实例。

### 2.2 统一数据总线 (Unified Data Bus)
- **实时镜像 (Real-time Snapshot)**：系统所有位号的最新值必须汇聚在 Redis 统一 Hash (`data_factory:v2:current`) 中。
- **多路复用**：不同引擎通过 `engine_id` 区分，但对外呈现为一个连通的位号空间。
- **高性能**：通过 Redis Pipeline 和 NumPy 优化，支持每秒万级以上的位号刷新率。

### 2.3 全局组态管理 (Config Server)
- **配置文件分离**：
  - **基础设施配置**：定义引擎实例、ID、数据源、存储周期等。
  - **业务逻辑配置**：基于 YAML 的 DSL 定义算法逻辑。
- **位号唯一性**：全局禁止不同引擎定义相同的位号。
- **命名空间隔离**：自动为各引擎分配命名空间，避免逻辑定义冲突。

### 2.4 可插拔消费者 (Consumers)
- **存储服务 (Storage)**：支持动态订阅所有引擎数据，并按设定的采样间隔持久化至 DuckDB。
- **协议转发 (OPC UA)**：支持将所有引擎的位号自动映射至 OPC UA 地址空间，供第三方监控软件调用。

## 3. 功能详细定义

### 3.1 基础设施编排
- 通过 `engines_manifest.yaml` 统筹全局。
- 支持引擎的动态发现与注册。

### 3.2 回放系统 (Playback)
- 支持多 Sheet 选择。
- 支持列名到标准位号名的映射。
- 支持循环播放。

### 3.3 诊断与可视化 (Observability)
- **架构拓扑图**：前端支持动态渲染系统各组件（引擎、中间层、消费者）的健康状态与链路关系。
- **实时性能分析**：展示各引擎的周期耗时（Exec Ratio）和位号更新频率。

## 4. 非功能需求

- **高性能**：算法模块（RANDOM 等）必须使用 NumPy 向量化或优化后的 Python 逻辑，减少 CPU 占用。
- **健壮性**：单个引擎崩溃不应影响整个数据总线和其他引擎的运行。
- **易用性**：提供“一键启动”能力，降低开发者搭建环境的成本。

## 5. 限制与约束
- 当前版本暂不保证跨机器的亚毫秒级时钟同步（基于本地 NTP 级同步即可）。
- DSL 中暂时不处理跨引擎的硬性循环引用。

## 6. Program 与数据模拟展示（2025-03 增补）

### 6.1 圆柱水箱与阀门级联
- **CYLINDRICAL_TANK**：入口改为外部传入体积流量 `inlet_flow`（m³/s），出口流量 `outlet_flow` 仍由托里拆利定律计算并对外暴露，支持「水箱接水箱」。
- **VALVE**：增加 `inlet_flow` 入参与 `outlet_flow` 输出，`outlet_flow = flow_coefficient * inlet_flow * 开度比例`。

### 6.2 默认展示与绘图缩放（`display_args`）
- 每个 Program 子类必须定义 `param_descriptions`（与 `stored_attributes` 配合）；**不再**在代码中维护 `display_attributes`。
- **DSL**：每条 `program` 可选 `display_args`（`attr` / `attr[ref]`），决定数据模拟页默认勾选列；`ref` **仅**影响预览图表纵坐标，**不得**改变快照或 CSV 数值。
- **数据模拟页**：默认列仅来自 DSL 非空 `display_args`（`display_variables`）；无前端位号勾选；图表可按 `plot_scales` 缩放，并可选用单 Y 轴或自动双 Y 轴。
- **运行态**：Redis/OPC UA 等仍发布全部 `stored_attributes`，不受展示配置影响。

### 6.3 接口
- `POST /simulate/preview` 返回 `display_variables`、`variable_meta`（含 `plot_scale_ref`）、`plot_scales`（供前端绘图）。
- `GET /templates/list`：扫描 `components/export_templates/templates/*.yaml` 返回模板名列表；新增模板文件后应自动出现在前端列表。
- `GET /export/format-defaults/{template_name}`：直接返回模板 YAML 中 `defaults`，字段与前端导出对话框完全一致（`header_rows`、`title_names`、`time_format`、`file_format`、`sheet_name`）。
- `POST /export/run`：支持可选 `selected_variables`（原始值，无 `ref` 缩放）；支持可选 `export_format`（与 YAML 解耦的表头与 `csv`/`xlsx`/`xls`）；无 `export_format` 时仍按 `template_name` 读 YAML 仅导出 CSV。`xlsx`/`xls` 响应使用 `file_content_base64`。

### 6.4 导出模板配置规范（2026-03）
- 每个默认模板必须使用单文件 YAML，位于 `components/export_templates/templates/`。
- 模板结构统一为：
  - `name: <template_name>`
  - `defaults: { header_rows, title_names, time_format, file_format, sheet_name }`
- 前端每次打开“数据生成”弹窗时，默认值由后端按当前模板 YAML 动态加载，不允许前端硬编码固定模板名。
- **（可选）独立合成数据生成**：提供 `generate_synthetic_export_file(tags, template, starttime, endtime, interval)`，不依赖仿真运行即可按模板输出 CSV/xlsx/xls，数据列为 100～199 锯齿波，便于格式与导入联调（见设计文档 5.7）。

### 6.5 PID 运行模式（MODE）（2026-03）
- **PID** 增加存储参数 **`MODE`**，默认 **`1`**。
- **`MODE == 1`**：每周期执行既有 PID 运算并更新 `MV`。
- **`MODE != 1`**：本周期不执行 PID 运算（不更新 `MV`、不累积积分、不推进微分状态）；文档与位号表须展示该参数说明。
