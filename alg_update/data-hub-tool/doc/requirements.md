# 需求

## 1. 背景

### 1.1 平台

数据中枢平台 **ibd-data-hub-web-v2.2**（基于 TDengine / IoTDB 等时序库）管理工业位号数据。每个"位号"（tag）有：
- 元数据：dataType（1~11）、单位、采集频率、所属组等
- 历史值：时序数据（tagTime, value, quality）

### 1.2 工具起源

`data-hub-tool/` 是新建的 Python 工具目录，对接上面这个平台。起源是 `alg_publish/`、`alg_sync/` 等算法工具项目（用 PyQt6 + 共享 `common_api.py`）。

最初项目只是 API 探针（`test_api.py`），后来演化成完整工具集。

## 2. 核心场景

### 2.1 历史值迁移（hisdata-migrate 主场景）

**用户痛点**：
- 平台不开放"导出再回灌"的官方工具
- 实际工作需要把一套环境的位号数据搬到另一套（比如开发 → 测试、备份 → 恢复）
- 平台 UI 操作几十个位号要几十次点击，效率极低

**工具解决**：
- 一行命令 / 一次点击
- 跨环境数据搬移
- UI 友好，能看到进度和验证结果

### 2.2 数据准备

**场景**：用户有外部数据源（CSV / xlsx / 数据库），要灌到平台

**工具解决**：
- `convert.py` + `xlsx_io.py`：把任意格式转成平台要求的 wide xlsx
- `migrate.py`（调 `common_api.import_tag_value_history()`）：批量上传

## 3. 用户角色

| 角色 | 使用方式 | 关键诉求 |
|---|---|---|
| **数据工程师**（命令习惯） | CLI 命令行 | 批处理、可脚本化、能看日志 |
| **运维 / 业务**（图形界面） | GUI | 看得见的进度、不用记参数、能中断 |
| **平台维护** | API 探针 | 调试、验证、抓包 |

`migrate.py` 一份代码两种 UI（CliUI / QtUI），都满足。

## 4. 功能需求

### 4.1 必须有

| 编号 | 需求 | 状态 |
|---|---|---|
| F1 | 读平台导出的 xlsx（long 格式） | ✅ |
| F2 | 转换成平台导入 xlsx（wide 格式） | ✅ |
| F3 | 调用平台 API 上传数据 | ✅ |
| F4 | 上传后验证数据是否落地 | ✅ |
| F5 | 跨环境支持（新 URL / 新用户） | ✅ |
| F6 | 友好的用户交互（CLI input / GUI 弹窗） | ✅ |
| F7 | 用户能决策"已有数据怎么处理" | ✅ |
| F8 | 打包成单文件 EXE | ✅ |

### 4.2 不做

| 编号 | 需求 | 不做的原因 |
|---|---|---|
| NF1 | 实时同步 / 定时任务 | 范围外，一次性导入为主 |
| NF2 | 数据库源（PostgreSQL / MySQL） | 当前只支持 xlsx / csv |
| NF3 | 数据转换 / 清洗（单位换算、异常值处理） | 是另一个工具的活 |
| NF4 | 位号元数据批量编辑 | 平台 UI 已支持，工具不重复 |
| NF5 | 删除位号 / 删除数据 | **当前不支持**，需要 delete API |
| NF6 | 大文件分片上传（>1GB） | 平台上限 1GB，工具默认 10000 点 / tag |

## 5. 非功能需求

| 类别 | 要求 |
|---|---|
| **可用性** | 一行命令启动 / 双击 EXE 启动；不需要查文档也能用基础功能 |
| **可移植** | 同一份代码能在多个 Python 环境（3.10+）跑；EXE 在无 Python 的 Windows 跑 |
| **可扩展** | 业务逻辑与 UI 分离，加新 UI（TUI / Web）不改业务代码 |
| **可观测** | 7 阶段日志输出，验证阶段打印每个 tag 的数据点数 |
| **健壮性** | 缺字段、tag 不存在、格式错、登录失败等都有明确错误信息 |
| **安全** | 不把 token 打到日志（自动截断打印只显示长度） |

## 6. 验收标准

1. ✅ 11 个 `aa_*` 标签，每个 10000 点，10s 周期，时间 2026-06-23 ~ 06-24
2. ✅ 数据值正确（数值 0~99 循环，BOOLEAN 0/1 交替）
3. ✅ 时间戳精确（每点 10s 间隔，无缺失无重复）
4. ✅ 命令行 + GUI 两种入口都跑得通
5. ✅ EXE 在没 Python 的 Windows 上能跑
6. ✅ 验证阶段能列出每个 tag 的实际数据点数

## 7. 用户故事

### US1：环境 A → 环境 B
> 作为数据工程师，我想把环境 A 的位号 `aa_double` 搬到环境 B，应该能：
> 1. 从 A 导出 xlsx（手动或脚本）
> 2. 跑 `migrate.py --xlsx aa_double.xlsx --target-url http://B:31501 ...`
> 3. 看到日志说"导入成功，requestId=..."
> 4. 在 B 的 UI 上看到数据

### US2：GUI 操作
> 作为运维，我想在另一台没 Python 的 Windows 上搬数据，应该能：
> 1. 把 `hisdata-migrate-v0.9.exe` 和 xlsx 拷过去
> 2. 双击 EXE
> 3. 选文件 → 填账号 → 点"开始"
> 4. 看进度条 / 日志 / 弹窗结果

### US3：批量决策
> 作为数据工程师，搬数据时部分 tag 已有数据，应该能：
> 1. 工具列出"已有数据的 tag"
> 2. 让我选每个 tag 是"覆盖"还是"跳过"还是"终止"
> 3. 按我的选择执行

## 8. 假设与约束

| 类型 | 内容 |
|---|---|
| **假设** | 目标环境的位号已经存在（由位号注册流程负责） |
| **假设** | 源数据 xlsx 格式与平台导出格式一致（4 列，每 tag 1 sheet） |
| **约束** | 平台 HTTP API 限制 1GB / 文件（远大于 10000 点 / tag） |
| **约束** | 导入是异步的（HTTP 200 ≠ 落地），必须用 `getHistoryValueFromDB` 回查 |
| **约束** | 时间方向、值类型平台宽容处理（V2/V3 验证过），但时间格式只支持 `yyyy/MM/dd` |
| **约束** | tagName filter 在 `/api/tag-info/page` 端不灵，必须本地过滤 |

## 9. 不在范围内（v0.9）

- 删除位号 / 删除数据（需要 delete API，平台已发现但未集成）
- 多线程 / 异步 I/O 优化（当前单 worker）
- 国际化（i18n）：界面只有中文
- 单元测试 / 集成测试套件（手动测试为主）
- CI/CD 配置
- Docker 镜像
- 远程日志 / 监控

## 10. 后续可能的方向

1. **delete API 集成**：让"覆盖"真的能清掉原数据
2. **CSV / 数据库源**：扩展到 `import_csv_tag_value_history` 和 DB
3. **Web UI**：基于 FastAPI / Flask 的远程管理界面
4. **定时同步**：用 cron 触发定期数据迁移
5. **数据校验规则**：导入前检查时间戳连续性、值范围等
6. **多 tag 批量进度**：当前 11 tag 看不到中间进度，大批量需要分批
