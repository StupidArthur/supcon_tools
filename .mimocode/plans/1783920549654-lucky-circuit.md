### 1.3 不放的事

- 根目录平铺 .go 业务文件 — 禁止(go-architecture §一)。
- 单 `App` 类型暴露全部方法 — 禁止(go-architecture §六 + §十三)。
- `bindings` 内调 `tptapi` 直接方法 — 禁止,必须经 `internal/rw.Service`。
- `internal/rw` 或 `internal/session` 任何文件 import Wails / `os/exec` / SQLite 驱动 — 禁止。
- 前端直接 import `wailsjs/go/...` — 禁止,必须经 `frontend/src/lib/api/*` 收口。
- 全局变量保存 context / client / Service — 禁止。
- `package common` / `helpers` / `utils` 收纳跨业务代码 — 禁止。
- Python Worker — 不引入。

### 1.4 模块拓扑(依赖方向)

```text
main → app/container → bindings + rw + session + platform
bindings → rw, session
rw → ports(tptapi.TptClient 由 app container 注入)
session → rw(共享同一 *tptapi.Service 实例)
```

`tptapi.Service` 实例在 `app/container.go` 创建一次,`Session` 与 `RW` 共享同一个句柄,避免双登录。`session.Service` 注入 `*tptapi.Service`(`rw.Service` 也持同一份),保证登录态只此一份。

### 1.5 关键类型(DTO — bindings 包,不暴露 tptapi)

```go
// bindings/dto.go
type LoginRequest struct {
    URL      string `json:"url"`
    Username string `json:"username"`
    Password string `json:"password"`
    TenantID string `json:"tenantId"`
}

type SessionInfo struct {
    LoggedIn bool   `json:"loggedIn"`
    URL      string `json:"url"`
    TenantID string `json:"tenantId"`
}

type DataSourceDTO struct {
    ID      int    `json:"id"`
    Name    string `json:"name"`
    Type    string `json:"type"`
    SubType string `json:"subType"`
    URL     string `json:"url"`
    Alive   bool   `json:"alive"`
}

type TagDTO struct {
    ID         int     `json:"id"`
    Name       string  `json:"name"`
    BaseName   string  `json:"baseName"`
    DataType   int     `json:"dataType"`
    TagType    int     `json:"tagType"`
    GroupID    string  `json:"groupId"`
    Quality    *string `json:"quality"`
    QualityCode *int   `json:"qualityCode"`
    DsID       int     `json:"dsId"`
}

type ReadRTRequest struct {
    TagNames []string `json:"tagNames"`
}

type RTValueDTO struct {
    TagName  string  `json:"tagName"`
    Value    string  `json:"value"`
    TagTime  string  `json:"tagTime"`
    AppTime  string  `json:"appTime"`
    Quality  int     `json:"quality"`
    DataType int     `json:"dataType"`
    DSID     int     `json:"dsId"`
    IsSuccess bool   `json:"isSuccess"`
    Message  string  `json:"message"`
}

type WriteRequest struct {
    Values map[string]any `json:"values"`        // tagName → 数值(任意 JSON)
    Confirm bool          `json:"confirm"`       // 防止误操作
}

type WriteResultDTO struct {
    Succeeded []string      `json:"succeeded"`
    Failures  map[string]string `json:"failures"`
    Readback  []RTValueDTO  `json:"readback,omitempty"`  // 写后 ~1s 回读
}

type ReadHistoryRequest struct {
    TagNames   []string `json:"tagNames"`
    BegTime    string   `json:"begTime"`     // "2026-01-01 00:00:00"
    EndTime    string   `json:"endTime"`
    Interval   int      `json:"interval"`
    IsSecond   bool     `json:"isSecond"`
    IsSource   bool     `json:"isSource"`
    Offset     int      `json:"offset"`
    Option     int      `json:"option"`
    Page       int      `json:"page"`
    PageSize   int      `json:"pageSize"`
    Sort       string   `json:"sort"`
    Mode       string   `json:"mode"`        // "page" / "fromdb"
}

type HistoryRowDTO struct {
    TagName string  `json:"tagName"`
    Value   string  `json:"value"`
    AppTime string  `json:"appTime"`
    Quality int     `json:"quality"`
}

type PublicError struct {
    Code    string `json:"code"`
    Message string `json:"message"`
    Kind    string `json:"kind"`  // "auth" | "api" | "http" | "parse"
}
```

### 1.6 错误模型(go-architecture §八 + wails-backend §十)

- Wails Binding 优先返回 `(T, error)`,前端 Promise 自然 reject。
- 内部错误:`tptapi.TptAPIError`(`model_full.go:134`)用 `errors.As` 提取 Code/Msg。
- `Kind == "auth"`(`code ∈ {A0230,A0201,A0202,A0203}` 或含 6 关键词)→ 前端弹回登录面板。
- 不返回 `Success bool + Error string` 风格(显式禁止)。

### 1.7 前端 API 收口(wails-backend §十三)

### 2.2 `internal/rw/service.go`

```go
    mu     sync.Mutex

func (s *Service) ReadHistory(ctx context.Context, q ListHistoryQuery) ([]model.HistoryRow, error)
```

`tptapi.TptClient` 字段通过 `ports.go` 定义:

```go
// internal/rw/ports.go
type TptClient interface {
    GetAllDsInfo(ctx context.Context) ([]model.DsInfo, error)
    QueryTagsWithQuality(ctx context.Context, p model.TagQueryParams) ([]model.TagRecordWithQuality, error)
    GetRTValue(ctx context.Context, names []string) ([]model.RtValuePoint, error)
    WriteTagValues(ctx context.Context, values map[string]any) error
    GetHistoryValue(ctx context.Context, p model.HistoryParams) (json.RawMessage, error)
    GetHistoryValueFromDB(ctx context.Context, p model.HistoryDBParams) (json.RawMessage, error)
}
```

实现 = `internal/adapters/tptapi/`(标准结构下放,轻量结构下与 `rw` 同包但独立写 adapter 方法),把 `*tptapi.Service` 的方法映射到此接口。

### 2.3 `internal/bindings/session.go` / `rw.go`(薄)

每个方法 ≈ 5 行:
1. `req` 校验
2. `if err := s.sessionSvc.EnsureLoggedIn(ctx); err != nil { return err }`(只对需要登录的方法)
3. `result, err := s.rwSvc.<Method>(ctx, ...)`
4. 转换 DTO
5. 返回

不写 SQL、不开新 client、不写业务规则。

### 2.4 `internal/app/container.go`

```go
func NewContainer() (*Container, error) {
    cfg, err := LoadConfig()
    if err != nil { return nil, err }

    logger := logging.New(cfg.LogLevel)
    tptSvc := tptapi.NewService()    // Service 空构造;首次 Login 才真正建立底层 TptClient

    sess := session.NewService(tptSvc, logger)
    rwSvc := rw.NewService(tptSvcAdapter(tptSvc), logger)

    return &Container{
        SessionBinding: bindings.NewSessionBinding(sess),
        RWBinding:      bindings.NewRWBinding(sess, rwSvc),
        Lifecycle:      app.NewLifecycle(tptSvc, logger),
    }, nil
}
```

### 2.6 前端(主要组件要点)

| 组件 | 关键交互 |

## 6. 等用户拍板 / 实施的剩余点

我列了几条非阻塞但建议拍板的项。在你确认前我不动手:

1. **轻量结构 vs 标准结构**:`internal/rw` + `internal/session` 共 2 个业务能力,刚好处在临界(lightweight 适用"只有 1~2 个业务能力")。我建议轻量(按上述布局)。
2. **`bindings` 拆分粒度**:`SessionBinding` + `RWBinding` 两个文件,各挂一片方法。Wails `Bind` 一次性挂两个对象。
3. **写后回读是否默认开启**:建议默认 1000ms,UI 显示 `written/readback/delta`,开关可关。
4. **`go.work` 还是 `replace`**:项目里没有 `go.work`,使用 `replace github.com/yzc/tpt_api => ../tpt_api/go`。
5. **`internal/platform/` 是否放文件**:本工具不需要系统对话框,我会留一个 stub 注释,方便以后扩展。

> 确认无误后我会退出 Plan 模式、按本计划逐步落实(先目录骨架 + bindings DTO + 单元测试 + 一个绑定 demo,再来真前端)。

## 11. 用户交互痛点(从当前 6 个组件读出,等会议讨论)

按"用户用起来会卡哪一步"扫描了一下当前组件,以下是我对当前 UI 的判断。先把这部分记入计划,再就优先级向你提问确认。

### 11.1 LoginPanel(`frontend/src/components/LoginPanel.tsx`)

- 默认 URL 写死 `https://supcontpt.supcon.com`、默认用户名 `admin`,真实用户多数不是这个站。但 placeholder 给的是 `https://host`,用户大概率看到 placeholder 会以为默认空 — **默认值与 placeholder 风格不一致**。
- "登录中…" 按钮文案用省略号,但页面其它按钮(`读历史` / `刷新实时值` / `写中…`)也是省略号 — 风格统一,不算问题。
- 当前 `info.loggedIn` 卡片只显示 URL + tenant,没有显示"登录到哪个用户"/"token 过期还剩多少"。用户没法判断这个登录还能撑多久。
- 没有"保存凭据"的 checkbox,而本仓库其它 GUI(`USER_MANAGER`)是支持 `LoginConfig` 的(`pid_debug_gui/go.mod:40` 提到过)。重新点开 GUI 要重输密码 — 麻烦。

### 11.2 TagSelector(`frontend/src/components/TagSelector.tsx`)

- 多选 checkbox + 列表一起,选中状态只有列表项里的 ☐ / ☒。没有"已选"汇总徽章、没有"清除选择"按钮。
- 触发 `onPicked(tags.filter(...))` 每个 toggle 都向 App 写入;RTReadPanel / WritePanel / HistoryPanel 三个面板都接受整个 `tags` 数组。**用户没有任何反馈说"当前操作的是哪 N 个位号"**,三个面板里没人复读这一信息。
- 数据源切换(`ds?.id` 变)时只清空 `tags`,没有提示"已切到 opcua-sim(9),加载中…"。DS 切换的瞬间用户看到一堆幽灵数据。
- "加载"按钮在用户敲完 keyword 后,需要先失去焦点再点 — 没有"敲回车就触发"的交互。

### 11.3 WritePanel(`frontend/src/components/WritePanel.tsx`)

- 单值写 + 单回读延迟输入。需求早期用户说"有时序窗口"——1s 是经验值,但 GUI 没有"先立刻读 RT 看现值 → 决定写什么 → 写 → 等 → 再读回"的连贯动作链。**用户如果想验证一个完整 round-trip,要在 WritePanel 和 RTReadPanel 间频繁切换**。
- `value` 解析规则(`/^-?\d+/`、`true`/`false`)是前端自定的语义,跟平台 `WriteTagValues` 接受的 `map[string]any` 是不是真的能识别这些,只能凭运气。
- 写值成功 / 失败的展示仅一行 "写入成功的位号: …" + JSON 失败信息。**回读和写入值差异对账(diff)** 没有突出 — 用户最关心的是"我写了 3.14,读回 3.14 / 3.139999 / NaN",这正是验证通路的重点。
- 一次只能写一个位号。用户当时说"读实时 / 写值 / 读历史" — 但如果要批量验证 5 个位号,得写 5 次。

### 11.4 HistoryPanel(`frontend/src/components/HistoryPanel.tsx`)

- 默认时间区间"过去 7 天到今天"硬编码。一些场景最近 7 天根本不是问题窗口。
- 时间输入是 `<input type="text">`(实际上 `Input` 组件没用 `type="datetime-local"`)。用户得自己手敲 `2026-07-13 12:00:00` 格式,差错率高。
- mode `fromdb / page` 区别很技术(`/getHistoryValueFromDB` vs `/getHistoryValue`),普通用户不知道什么时候该选谁。
- "interval (page 模式专用)" + "每页条数" 这些参数直接暴露给前端,但只有 mode=page 时 interval 才生效 — 没有条件禁用,用户可能误填产生无意义结果。
- **关键缺位**:历史表里没有"appTime 与刚才写入时刻对齐"的标注 — 用户写完 3.14 后读历史,看不到"我刚写的那条记录在哪?"绿色高亮、闪烁、scrollIntoView 都没有。

### 11.5 RTReadPanel(`frontend/src/components/RTReadPanel.tsx`)

- 刷新一次就显示,但**没有"自动轮询"**。用户想看 ~1s 反映,~4s 历史落库,只能反复手点。
- 没有"现在显示的实时值"与"上次刷新时值"的列对比 diff。
- 品质码 `q` 直接显示成整数,普通用户不知道 0=Good / 192=Bad / 128=Uncertain 等。需要一个 status 翻译 + 颜色。
- 选中位号改了,**已读到的旧 rows 不动**,用户可能不知道这是"上一次手刷的结果,跟现在选中位号无关"。

### 11.6 SourceSelector(`frontend/src/components/SourceSelector.tsx`)

- select 显示 `{name} (id={id}, sub={subType}, {alive ? 'alive' : 'dead'})`。`dsSubType` 是平台枚举 `4=OpcUa` `5=Mysql` 等,但显示成数字,普通用户看不懂。
- 切 DS 后没有重置其它面板的内存里已经读到的数据,只是把传入 `TagSelector` 的 `ds` 改了。

### 11.7 App 顶层布局(`frontend/src/App.tsx`)

- 6 个 panel 单列排,自上而下:登录 → 数据源 + 位号(2 列)→ RTRead → Write → History。窗口 1280 宽时右侧大片空白。
- **完全没有错误显示区**。任何 binding 抛错,只能在触发它的面板内看到红字;用户切到其它面板,错误就消失。
- `setTags` 是顶层 state,所有面板读这份 — 写值结束后有没有改动 RTRead?改动没 — `WritePanel` 写完不通知 `RTReadPanel` 刷新。
- Session 唯一,不响应"token 过期自动跳回登录"。`Service.Client()` 在 5min 后会自动重登(`state_full.go:49`),但前端没有"401 → 弹 Login"的反馈。

### 11.8 工程层面的 UX 反馈回路(不只是组件)

- **没有"事件总线"机制**:WritePanel 写完 → RTReadPanel 自动刷 → HistoryPanel 自动滚到刚才那条 → 用户看到完整闭环。当前状态是 3 个独立按钮各干各的,用户得来回点 3 次。
- **没有 toast / 顶部全局栏**:bind 抛 `PublicError{Kind:"auth"}` 的反应 = 用户看到某面板里一行红字,瞬间就被新操作覆盖。
- **持久化空白**:用户多次重启,默认 URL、用户名、最近一次选过的 DS、最近一次写过的值 / 历史范围,全都没记忆。

### 11.9 与平台事实相关的两点(我读了源码发现的"用户不知道会踩"的事)

- `WriteTagValues` 同步写后,`GetRTValue` 需要 ~1s 反映,`QueryHistoryValue(is_source=True)` +0s 立即命中但用普通 `getHistoryValueFromDB` +0s 未必能取到刚写的那条(平台有 ~5s 轮询延迟)。**GUI 默认的 1000ms 回读延迟已经踩对了点**,但用户写值结束后立刻读历史,他不知道要等 ~4s,这是数据通路验证的核心 target。
- `QueryHistoryValue(is_source=True)` 与普通 `getHistoryValueFromDB` 走的逻辑不一样 — 经验上写后 +0s `is_source=true` 可查到,但这前提是 mode=page 才传得上;mode=fromdb 不支持 `isSource`。**而当前面板 mode 描述没把这个差别说清楚**。

### 11.10 优先级候选(用于会议排序)

我现在按"对核心目标 — 数据通路验证 — 的关键程度"排了一下,带价格感:

## 13. 用户拍板:6→2 整合方案(详细设计)

### 13.1 整体形态(确认口径)

把当前 6 个 panel(`LoginPanel` / `SourceSelector` / `TagSelector` / `RTReadPanel` / `WritePanel` / `HistoryPanel`)重组成**两个区块** — **登录区 + 数据验证区**。

**两个区块的定义**(用户确认):
- 登录区:更紧凑,两行布局
- 数据验证区:选 DS → 选位号 → 选中后才出现"实时 / 写值 / 历史"3 块

**用户对各子题的拍板**:
1. **位号选择 = 单选**。一次只验证 1 个位号。
2. **实时刷新 = 选中后才开始**(没有选中时不轮询)。
3. **历史时间刷新** = 独立的"刷新时间"按钮(改 start / end,只动时间区间),**不**带任何查询动作;**查询动作是另外的"查询历史"按钮**。"刷新时间"按钮的本意是把 end 推到 now、再算 start。
4. **写值不接受只读约束**,只读位的写值失败靠 toast 报给用户。

### 13.2 登录区(LoginPanel 紧凑版)

两行布局:
- **第 1 行**:`URL`(占 ~70%宽) + `租户 ID`(占 ~30%宽)
- **第 2 行**:`用户名`(占 ~33%) + `密码`(占 ~33%) + `登录` 按钮(占 ~33%)
- 默认 URL = `http://10.10.58.153:31501`
- 默认用户名 / 密码 空(用户口述是这意思 — 默认就没意义,只能占位)
- 默认 tenant 空(单租户)
- 错误显示一行红字在两块之下(整体宽度)

不再显示"已登录"卡片(整合后,数据区 disabled 就表明未登录)。

### 13.3 数据验证区(新组合 — 一个组件 + 状态机)

#### 13.3.1 上下文部分(顶部)

- **数据源 select**:从 `rwApi.listDataSources()` 拉,展示 `{name} ({alive ? 'alive' : 'dead'})`。
  - 默认 selected = 第一条
  - 选完后 → 触发 `rwApi.listTags({dsId})` 拉位号
- **位号 select**:
  - 拉完位号之后显示
  - **单选**;用户口述"只能验证 1 个"
  - 旁边的 input = 模糊关键字(可过滤列表)
- **未登录 / 未选 DS / 未选位号** 时:
  - 未登录:整个数据区半透明,提示"请先登录"
  - 未选 DS:提示"请选择数据源"
  - 未选位号:提示"请选择位号"

#### 13.3.2 位号选中后下方三块并排

把当前 `RTReadPanel` + `WritePanel` + `HistoryPanel` **合并**为一个组件 `VerifyPanel.tsx`,内含三列并排 / 紧凑型卡片:

- **[A] 实时值卡**
  - **选中位号后**才开始 1 秒级轮询(默认 1000ms)
  - 显示:`tagName | value | tagTime | quality`
  - 顶部小图标表示是否在轮询(空心 / 实心 + 频率)
  - 不需要"手动刷新"按钮(轮询自动)
  - 轮询通过 setInterval + `rwApi.readRealtime([tagName])`,组件 unmount / 切位号时清 interval

- **[B] 写值卡**
  - 一个 input + 一个提交按钮
  - input 旁边显示当前位号名只读(明确"这是写给哪位的")
  - 默认值预填当前 RT 值(`A 卡的 value`)
  - 不做前端类型 parse(number / true-false)- 直接当字符串传给后端(map[string]any 用 raw string 让平台自己挑)
    - 备注:目前原 `WritePanel` 是按正则 parse 后再发数字,这里按用户意图"简化"。**这一项是否真这样实施要在 plan 用例里记一笔,需要用户再拍一次。**
  - 提交:
    - 调 `rwApi.writeValues({values: {[tagName]: value}, readbackDelayMs: 1000})`
    - 写值成功 / 失败用 **toast**(顶栏右下角)统一报
    - 写完 1 秒后回读自动落到 `A 卡`,用户视觉上"实时值的 value 变了"
    - 不再单独渲染回读表 — `[A] 卡` 就是回读
  - 提交瞬间 `B 卡` 按钮 disable 并显示 "写中…"

- **[C] 历史值卡**
  - 默认时间 = **过去 1 分钟**(`start = now - 60s`, `end = now`)
  - 时间显示:**mm:ss**(分钟:秒)— 因为窗口 1 分钟很短,日期没意义。平台接受 `yyyy-MM-dd HH:mm:ss`,前端拼好再传。
  - 用户可通过"刷新时间"按钮(独立)把 `end` 推到点击时刻,把 `start = end - 1min`(用户口述"刷新时间,是刷新时间设置的 start 和 end",不过我没确定是否真的 start = end - 1min 还是别的方式 — 见 §13.4)
  - 查询是另一个独立按钮"查询历史",按当前 start / end 调 `rwApi.readHistory({...})`
  - 查询结果:表格 `tagName | value | appTime | quality`
  - 默认 `mode = fromdb`(不动 mode 切换)
  - 不显示 pageSize / interval(简化掉 — 多分页能力是过度设计)

#### 13.3.3 Toast(顶栏右下角)

- 全局错误展示。组件接 `ErrorBus` (App 提供的 context)。
- 三类:`info`(蓝色)/ `success`(绿)/ `error`(红)
- 每个 toast 5 秒自动消失;`error` 可点击 X 手动关闭
- 写值成功 / 失败 / 登录失败 / 读 RT 失败 / 读历史失败都进 toast
- `A 卡`的实时值下方,如果有当前位号的"上次写值 vs 写后 RT"两个值不一致,提示"⚠️ 与上次写入不一致"的 toast(此条暂作可选 — 不在 §13.1 用户拍板范围)

### 13.4 我还未拍板的几个细节(实现前再问你)

用户在回答历史刷新按钮时说"刷新时间,是刷新时间设置的 start 和 end,查询是另外的按钮"。我理解:
- 按钮 A = "刷新时间" → 只更新 start / end 显示(推 end 到 now,start = end - 1min),**不**触发任何网络请求
- 按钮 B = "查询历史" → 用当前 start / end 调 `rwApi.readHistory(...)`,**不**刷新时间

但有一个细节我还不确定 — 你之前用"过去 1 分钟到当前"做默认;现在"刷新时间"按钮是不是同一种语义(**写死 start = end - 1min**)还是允许用户调窗口宽度?实现差很大:
- (i) 写死 1min 窗口:实现简单,UI 上"刷新时间"按钮就是这个意思。
- (ii) 允许用户扩大窗口:加一个"窗口宽度"input,start = end - width。

我先按 (i) 实现(写死 1min)做完第一版,你试一下,觉得不够灵活再改 (ii)。

另一个仍未拍板的事:**B 卡(input 提交按钮)** — 是否按 §13.3.2 描述那样"input 默认填入 A 卡的当前 value"?如果你想让"按 RT 改一点点"的工作流成立,这是对的;如果你想让"input 永远是空,自己填",这又是另一个方向。**第一版我按"默认填入 A 卡"实现,你试一下,觉得太自动再改**。

第三个未拍板的:**写值后是否需要"历史卡自动跑一次查询"**——按你 §13.1 拍板,查询是单独按钮 + 实时刷新选中才开始,没有要求历史卡写后自动查。**所以写后历史不动**;用户想看历史落库得点"查询历史"。**这是你的拍板的口径**。
