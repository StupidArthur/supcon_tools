## 2. 核心技术路线：Go 壳 + 托管 Python 服务（路线 B，用户选定）

**用户决策（2026-07-22）**：OPC UA Server 没有必要用 Go 重写，复用成熟的 asyncua 引擎。

### 架构

```
┌─────────────────────────────────────────┐
│ GUI exe（Wails + Go，≤15MB）              │
│  ├─ 前端：状态/启停/参数/节点列表          │
│  ├─ app.go 绑定层                        │
│  └─ 管控层：生成 YAML → 拉起/停止服务进程   │
│         │ 解析 stdout 启动标记             │
│         ▼                                │
│ 服务 exe（现有 main.py 的 PyInstaller 产物）│
│  └─ asyncua OPC UA Server（26 节点）      │
└─────────────────────────────────────────┘
```

- **优点**：
  - 协议层零风险——asyncua 成熟，现有引擎（节点构建、变化引擎、可写时间戳修复）原样复用
  - 不依赖实验性库（gopcua server 仍为 experimental，降级为备选，见路线 A）
  - 现有仓库已有 `main.spec` / `ua_mocker.spec` 打包配置，服务 exe 产物现成
- **代价（明示，用户已知悉并接受）**：
  - 分发形态为**双 exe**：GUI exe（≤15MB，仍守红线）+ 服务 exe（PyInstaller 约 30MB）
  - 完整分发包约 40~50MB，超出 skill 的「单文件 ≤15MB」口径——**作为经用户批准的例外记录在案**，
    GUI 自身体积仍按 ≤15MB 验收
- **路线 A（纯 Go / gopcua）保留为备选**：若日后要求单文件分发，可切回

### 对现有代码的边界（零侵入）

- `main.py` / `server_main.py` 等**一行不改**
- 集成契约 = 现有稳定的控制台输出（`server_main.py` docstring 已钉死）：
  `开始构建节点树` → `构建完成` → `服务启动成功 opc.tcp://host:port/ua_mocker/` → `节点数量: N` → `designed by yzc`
- GUI 靠解析这些标记判断启动成功、提取 endpoint 与节点数；进程提前退出 = 启动失败，
  读取服务日志文件（`ua_mocker_YYYYMMDD.log`）尾部作为错误信息展示

## 3. 模块拆分

```
gui/
├── main.go             Wails 入口 + 窗口配置（1180×780，白底，最小 920×620）+ OnShutdown 兜底停服
├── app.go              绑定层：暴露给前端的 6 个方法（透传，不含业务逻辑）
├── nodespec.go         内置组态模型：13 类型表 + 26 节点规格（供 YAML 生成与列表展示共用）
├── yamlgen.go          由节点规格 + 端口/cycle 生成组态 YAML（文本模板，不引第三方 YAML 库亦可）
├── launcher.go         服务进程管控：定位服务 exe/python、spawn、stdout 标记解析、
│                       超时/失败判定、停止（Windows: taskkill /T /F）、存活探测
├── nodespec_test.go    组态规格单元测试（26 条、类型完整、r/w 成对）
├── yamlgen_test.go     YAML 生成单元测试（端口/cycle 注入、default 正确、可被 PyYAML 等价解析）
├── launcher_test.go    标记解析/失败判定等纯逻辑单元测试
├── doc/                需求/设计文档
└── frontend/           共享前端层
    └── src/
        ├── lib/api.ts              Wails 绑定统一封装
        ├── components/ui/          Shadcn（button/card/badge/input/scroll-area/tooltip）
        ├── components/             StatusBar、SettingsForm、NodeTable、Toast
        ├── App.tsx                 状态上提 + ToastProvider + 三态渲染
        └── style.css               Tailwind 指令 + Notion 设计 Token
```

**分层纪律**：`nodespec.go` / `yamlgen.go` / `launcher.go` 不 import Wails，可独立 `go test`；
`app.go` 只做绑定透传；错误走结果结构体 `Error` 字段，不 panic。

### 服务程序定位规则（launcher.go）

按序查找，命中即用：
1. GUI exe 同目录下的 `ua_mocker.exe`（**分发形态**：双 exe 并排）
2. GUI exe 同目录下的 `main.py` + 系统 `python`（开发形态之一）
3. 仓库开发形态：相对 GUI exe 的 `../../../main.py`（`gui/build/bin/` → 仓库根）

均未命中 → 返回明确错误：「未找到服务程序，请将 ua_mocker.exe 置于本程序同目录」。

## 4. 对外接口（Go → 前端绑定）

| 方法 | 入参 | 返回 | 说明 |
|------|------|------|------|
| `GetSettings()` | — | `{port, cycleMs}` | 当前参数（默认 18955 / 1000） |
| `SetSettings(port, cycleMs)` | int, int | `{ok, error}` | 仅停止状态可改；端口 1024~65535 校验 |
| `StartServer()` | — | `{ok, endpoint, nodeCount, error}` | 生成 YAML→spawn→等标记（超时 20s）；幂等 |
| `StopServer()` | — | `{ok, error}` | taskkill 进程树；幂等 |
| `GetServerStatus()` | — | `{running, endpoint, nodeCount, pid}` | 进程存活探测 + 缓存信息 |
| `ListNodes()` | — | `[]{nodeId, type, mode, default}` | 固定 26 条（本地规格，不连真实服务） |

## 5. 界面设计

```
┌──────────────────────────────────────────────┐
│ ● 运行中   opc.tcp://0.0.0.0:18955/ua_mocker/ │  ← 顶栏：状态徽标 + endpoint + [复制]
│                              [停止] [启动]    │
├──────────────────────────────────────────────┤
│ 端口 [18955]   周期 [1000] ms   （停止时可编辑）│  ← 参数区
├──────────────────────────────────────────────┤
│ 节点列表（26）                                 │
│ NodeId            类型      模式      默认值   │  ← ScrollArea 表格
│ ua2_boolean_r_1   Boolean   自变化    —       │
│ ua2_boolean_w_1   Boolean   可写      false   │
│ ...                                          │
└──────────────────────────────────────────────┘
```

- 状态徽标：运行中绿点 / 已停止灰点；启动失败 Toast 红色提示（含服务日志尾部错误）
- Notion Token（暖灰底 + 珊瑚橙强调），系统字体，零外部资源
- 节点列表为静态展示（固定组态），实时值验证按需求用外部客户端

## 6. 关键实现逻辑

1. **组态生成**（`yamlgen.go`）：由 `nodespec.go` 的 26 条规格 + 当前 port/cycle 渲染 YAML 文本，
   写到 GUI exe 目录下 `ua_types_gui.yaml`；命名 `ua2_<type>_r_1`/`ua2_<type>_w_1`，ns=2，
   各类型 default 与 ua2_types.yaml 一致（false / 0 / 0.0 / "" / 2025-01-01T00:00:00Z）
2. **启动**（`launcher.go`）：`exec.Command(服务程序, yaml路径)`，pipe stdout，
   逐行匹配 `服务启动成功` 提取 endpoint、`节点数量` 提取 N；20s 内未见标记且进程退出 → 失败，
   读 `ua_mocker_YYYYMMDD.log`（服务 exe 同目录）尾部返回前端
3. **停止**：`taskkill /T /F /PID <pid>`（Windows；服务无持久状态，强杀安全），
   非 Windows 回退 `Process.Kill`；窗口 `OnShutdown` 钩子兜底调用
4. **参数互斥**：运行中 `SetSettings` 直接返回 error（前端同时禁用输入框）

## 7. 横切规则检查

- **运行安全与运维**：服务无持久化数据，无丢数据风险；落地三项——关窗释放进程（OnShutdown 兜底）、
  启停幂等、启动失败错误可见（Toast + 日志尾部）
- **LLM 集成**：不涉及

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 控制台输出契约变化导致解析失效 | 契约在 server_main.py docstring 中已钉死；解析失败有超时兜底与日志回读 |
| 服务 exe 未放置/缺失 | 定位规则三级回退 + 明确错误文案指引 |
| 端口占用 | 服务进程启动即退出，GUI 读日志尾部（asyncua 报错）Toast 展示 |
| PyInstaller 服务 exe 体积大 | 已接受的例外；GUI 自身仍 ≤15MB |
| 杀进程树不彻底 | taskkill /T 按进程树杀；停止后探测端口/进程确认 |
