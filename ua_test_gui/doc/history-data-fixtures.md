# 历史数据制造指南

本文档定义 UA-3 中历史查询、历史落库、响应时间和性能测试的统一造数方法。

历史相关用例必须明确引用本文档的方式 A、B 或 C，不得只写“准备历史数据”。

## 1. 选择原则

| 方式 | 数据来源 | 主要用途 | 不适合验证 |
|---|---|---|---|
| A：自动采集 | asyncua 修改 OPC UA Mock 节点，由 DataHub 自动采集 | 采集链路、频率、断线恢复、删除恢复后的历史连续性 | 精确分页数量、精确边界和填充规则 |
| B：历史导入 | `importTagValue`，大数据量时使用 Excel/ZIP `importTagValueHistory` | 时间边界、分页、排序、interval、option、offset、多位号隔离、历史查询性能 | OPC UA 自动采集链路 |
| C：实时写入 | `writeTagValues` 写唯一值，再查询历史 | 回写记录、tagTime、qualityCode、isSource、写后历史 | 纯自动采集语义 |

同一用例只验证一个主要目标。需要端到端证明时使用 A；需要精确数据集时使用 B；需要验证写入与历史关系时使用 C。

## 2. 公共命名和隔离

每次运行生成唯一 `runId`，建议格式：`YYYYMMDD_HHMMSS_<random>`。

- 位号名：`ua34_{runId}_{purpose}_{index}`；
- 值域按用途隔离：
  - A：`310000 + sequence`；
  - B 单位号：`340000 + sequence`；
  - B 多位号：A 使用 `410000 + sequence`，B 使用 `420000 + sequence`；
  - C：`510000 + sequence`；
- 每次运行使用独立时间窗口；
- 测试记录保存 `runId`、位号 ID、数据源 ID、时间窗口、预期点集合和实际点集合。

历史记录通常不能逐条删除，因此查询必须同时限定本次位号名和本次时间窗口。

## 3. 方式 A：自动采集造数

### 3.1 适用用例

- 自动采集是否进入历史；
- frequency 对采集时间间隔的影响；
- 数据源断线、恢复前后的历史；
- 位号软删除、恢复前后的历史；
- OPC UA 源值、DataHub 实时值和历史值的一致性。

### 3.2 前置条件

1. 使用可写、静态且不会自行变化的 Mock 节点；
2. 创建一次位号，`dataType` 与源节点一致；
3. 等待 `getRTValue` 返回有效质量；
4. 记录开始时间 `T0`。

### 3.3 操作步骤

对节点按顺序写入唯一值，例如：

| sequence | 源端值 |
|---:|---:|
| 1 | 310001 |
| 2 | 310002 |
| 3 | 310003 |
| 4 | 310004 |

每次写入执行：

1. 通过 asyncua 写源节点；
2. 记录客户端单调时间和 UTC 时间；
3. 轮询 `getRTValue(isFromDB=false)`，直到出现目标值或超时；
4. 记录首次实时可见时间；
5. 至少等待一个配置采集周期后再写下一个值。

全部写入后，等待 `historyVisibilityTimeout`，查询覆盖 `T0` 至结束时间的历史窗口。

### 3.4 断言原则

- 关键唯一值必须出现在历史中；
- 历史点必须属于目标位号和时间窗口；
- 自动采集记录数量不能机械要求等于运行秒数；
- frequency 用例比较相邻历史点时间间隔的分布和容差；
- 断线期间不得出现伪造的新值；
- 恢复后应重新出现新历史点。

## 4. 方式 B：历史导入造数

### 4.1 适用用例

- 起止时间边界；
- 分页、排序、全集完整性；
- `interval`、`option`、`offset`；
- 多位号混合查询；
- 13 种类型的历史表示；
- 响应时间和历史查询性能。

历史导入在这些用例中属于测试夹具，不是被测功能。导入接口成功不代表数据已落地。

### 4.2 规则数据集

为单个位号生成固定间隔数据，例如 25 个点：

- 起始时间：`T0`；
- 间隔：10 秒；
- 第 i 点：`tagTime = T0 + i*10s`；
- `tagValue = 340000 + i`；
- `quality = 192`；
- i 范围：0～24。

该数据集可用于：

- `pageSize=10` 的三页查询；
- 升序和降序；
- 普通时间窗口；
- interval=20 秒等规则采样。

### 4.3 边界数据集

在查询开始时间、结束时间以及边界前后各放一个点：

- `begin - 1s`；
- `begin`；
- `end`；
- `end + 1s`。

通过唯一值判断接口采用闭区间、开区间或半开区间。

### 4.4 稀疏数据集

用于填充和偏移规则：

- `T0 + 3s -> 350003`；
- `T0 + 17s -> 350017`；
- `T0 + 44s -> 350044`。

使用 10 秒采样间隔查询，验证前向采样、不填充和 offset 行为。

### 4.5 多位号数据集

- tag_A：`410000～410099`；
- tag_B：`420000～420099`。

两个位号可以使用相同时间戳，但不得使用相同值域。任何串位号都能直接识别。

### 4.6 导入方式

常规确定数据使用 `importTagValue` JSON 导入。每条数据至少包含：

```json
{
  "tagName": "ua34_<runId>_page_01",
  "tagValue": 340001,
  "quality": 192,
  "tagTime": "2026-07-11 10:00:10",
  "appTime": "2026-07-11 10:00:10"
}
```

大数据量性能测试使用 Excel/ZIP `importTagValueHistory`，避免通过大量单次请求造数。

### 4.7 造数完成判定

导入后轮询历史查询，只有同时满足以下条件才允许开始正式用例：

1. 目标位号存在；
2. 记录数量达到预期；
3. 首尾唯一值存在；
4. 最早和最晚时间处于预期范围；
5. 不包含其他 runId 的数据；
6. 多位号值域没有串位号。

超时未满足时，将用例标记为 `setup_failed`，不得判定历史查询功能失败。

## 5. 方式 C：实时写入造数

### 5.1 适用用例

- 写入后是否进入历史；
- 指定 `tagTime`；
- 指定 `qualityCode`；
- `isSource` 是否包含回写记录；
- 批量写入后的历史完整性。

### 5.2 操作步骤

1. 创建专用可写位号并保存原值；
2. 调用 `writeTagValues` 写入 `510000 + sequence`；
3. 每次写入使用唯一值，必要时指定 `tagTime` 和 `qualityCode`；
4. 检查返回 `tagNames` 和 `failMsg`；
5. 查询实时库确认写入结果；
6. 查询数据库实时值确认最终落地；
7. 轮询历史窗口直到目标值可见或超时。

### 5.3 断言原则

- 只有返回成功并且读回成功的值才计入预期历史；
- 历史值、时间和质量应符合接口实际语义；
- `writeTagValues` 是否回写 OPC UA 源端需要 asyncua 独立确认；
- 写入失败项不得出现在历史中。

## 6. 性能数据集

历史查询性能使用方式 B：

- 专用位号和专用时间窗口；
- 数据量按阶段准备，例如每位号 1千、1万、10万点；
- 多位号场景保持不同值域；
- 每阶段造数完成后记录真实点数；
- 正式压测期间不再导入数据，避免导入任务干扰查询结果；
- 压测后抽样检查首、中、尾记录以及分页合集。

## 7. 清理

1. 删除本次创建的位号配置；
2. 恢复 OPC UA 源节点原值；
3. 保存 runId 和遗留历史时间窗口；
4. 不假定删除位号会同步清除历史；
5. 由测试环境维护任务定期清理历史库中的测试数据。
