# 测试用例设计方法论

本文件归纳测试用例评审过程中形成的设计思路与规则，指导后续用例编写。

---

## 1. 层次划分

### 1.1 一级点：同抽象级、不混维度

- 功能测试和非功能测试分开成章，不混在同一层
- 一级点之间是平等的能力域或测试维度，不存在包含关系
- 划分逻辑按被测系统的能力递进，前一个是后一个的前置条件
- 示例：UA 客户端 -> 数据源管理 -> 位号管理 -> 数据采集 -> 数据回写（功能）；系统响应性 -> 读写吞吐与容量（非功能）

### 1.2 二级点：覆盖完整生命周期

- 每个一级点下按生命周期或操作类型拆分
- 覆盖从创建到销毁的全过程：建立 -> 启停 -> 故障恢复 -> 并存 -> 删除
- 不遗漏生命周期环节

### 1.3 三级点：一条用例一个测试意图，可含多个断言

- 一个测试动作 + 多个断言 = 一条用例，不要拆成多条
- 示例：断开后 alive/quality/值/历史 4 个指标的变化延迟，是一条用例 4 个断言，不是 4 条用例

---

## 2. 前置条件与验证逻辑

### 2.1 测"停"之前先证"动"

- 验证某操作导致"数据停止变化"时，同一条用例必须先证明"数据正在变化"
- 示例：测禁用数据源后历史不再增长 -> 先取 2 次历史确认条数在增加 -> 再禁用 -> 再验证不再增长

### 2.2 数据源配置与测试意图匹配

- 测"值不变"用 change=true（值持续变化）的数据源
- 测"值固定"用 change=false 的数据源
- 前置条件中的 mock 配置必须与测试意图一致

### 2.3 前置条件可能无法满足的用例

- 保留用例，不删除
- 执行后若前置条件无法满足，标记 NA（Not Applicable）
- 示例：测"删除有位号的数据源后位号 RT 状态"，如果系统不允许删除有位号的数据源，则此用例 NA

---

## 3. 覆盖策略

### 3.1 多因素矩阵覆盖

- 存在多个独立二元因素时，用矩阵确保全覆盖
- 示例：数据源删除 = 禁用/启用 × 有位号/无位号 = 4 条用例

### 3.2 边界值必测

- 每种数据类型测：默认值、测试值、上限值、下限值、超范围值、类型不匹配值
- 各类型边界值列出参考表

### 3.3 重复测试收集统计值

- 时延类测试不能只测 1 次
- 重复 N 轮（如 5 轮），输出平均值、最大值、最小值
- 验证多轮数据一致性

---

## 4. 被测功能与验证工具的区分

### 4.1 API 本身可能是被测功能

- 某些 API 既是验证工具，本身也是系统功能
- `ds-info/test`（枚举/读/写/历史测试）本身就是数据源管理的功能，需要有自己的测试用例
- 不能拿被测功能当验证工具自己验自己

### 4.2 用例中明确"测什么"和"用什么"

- 每条用例的"验证手段"字段要区分：
  - **测试对象**：本条用例要验证的系统行为
  - **验证工具**：用来读取/对照的 API 或手段（它自身的正确性由其他用例覆盖）
- 如果用 API A 验证功能 B，则 API A 必须有独立的用例覆盖其自身正确性

### 4.3 替换验证 API 时的检查

- 当用新 API 替换旧 API 做验证时，确认旧 API 的功能在其他用例中有覆盖
- 不能因为换了验证手段就遗漏对原 API 的测试

---

## 5. 用例详细度要求

### 5.1 结构化字段

每条用例必须包含：
- 编号
- 三级点（用例名）
- 前置条件（含 mock 配置、数据源状态、位号状态）
- 测试步骤（可执行的有序步骤）
- 预期结果（具体的可判定结果，不是模糊描述）
- 验证手段（具体调哪个 API、读什么字段）

### 5.2 预期结果要可判定

- 不写"值正确"，写"RT 值 = mock 节点当前值"
- 不写"采集正常"，写"getRTValue 返回 quality=192 且值在变化"
- 不写"系统不崩溃"，写"API 不报错、alive=false、进程存活"

### 5.3 理解系统实际行为

- 设计用例前先了解系统的实际机制（如位号删除有两级：软删进回收站 + 物理删）
- 用例要反映系统真实的交互逻辑，不是凭空假设
- 不确定的行为设计为探索性用例，测过再定
