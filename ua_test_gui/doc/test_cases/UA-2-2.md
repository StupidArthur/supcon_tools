# UA-2-2 位号查询

验证用户能否准确找到目标位号、查看配置与运行状态，并在数据变化后获得最新查询结果。

本文件按**用户任务和业务场景**组织，不按 API 拆分。API 只用于构造测试数据、执行等价操作或独立核验。

---

## 设计约定

### 功能边界

本二级点覆盖：

1. 查看当前可管理的位号列表；
2. 按系统位号名、底层位号名、数据源和分组查找；
3. 使用组合条件缩小结果范围；
4. 查看配置、实时值、质量码和更新时间；
5. 浏览数据源中可注册的底层节点；
6. 在大量结果中分页、排序和刷新；
7. 配置及运行状态变化后，查询结果正确更新。

### 执行原则

- GUI 有对应能力时，优先按用户路径执行；API 用于独立核验。
- GUI 暂无对应能力时，可用等价 API 执行，但用例仍按功能命名。
- 配置字段以 `tag-info/page` 为核验来源；实时值、质量和时间以 `queryWithQuality`、`getRTValue` 为核验来源；源端值以 asyncua 为独立对照。
- 浏览节点时，正式“未注册集合”由 `dsId + tagBaseName` 与已注册集合做差得到，不能完全信任接口的未导入过滤。
- 临时数据使用 `ua22_{runId}_` 前缀并在用例结束后清理。

### 基础测试数据

- 数据源 A、B 均运行；A 有 12 个已注册位号，B 有 3 个；
- 两个数据源至少存在一个相同 `tagBaseName`；
- A 中至少 20 个未注册节点；
- 至少两个普通分组，一个空分组；
- 至少一个变化节点、一个静态节点、一个软删除位号和一个收藏位号。

## 执行与清理要求

1. 每次运行生成唯一前缀 `ua22_{runId}_`；
2. 用例前记录目标结果集合，断言时优先比较稳定 ID 集合，不比较列表位置；
3. 涉及动态值时记录查询时间窗口和允许刷新容差；
4. 数据源断线、分组、收藏和编辑操作结束后恢复原状态；
5. 新增位号结束后删除，写值时恢复 UA 源端原值；
6. 清理失败单独记录为 `cleanup_failed`，不得覆盖原测试结论；
7. 探索用例保存完整条件、页面结果、API 请求响应和最终集合，规则确认后再转为回归。
