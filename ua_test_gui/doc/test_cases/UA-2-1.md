# UA-2-1 位号新增

验证单个或批量新增位号时，配置是否正确保存，并通过实时读取、源端回写或历史查询证明位号实际可用。

本文件允许与数据采集、数据回写和历史数据能力交叉；这些读写操作在本二级点中用于证明“新增成功形成闭环”，不替代后续专项测试。

---

## 用例约定

### 编号与名称

- 编号使用稳定数字 ID：`UA-2-1-001`、`UA-2-1-002`……
- 三级点名称使用 `area_title`：例如 `数据源_已组态运行`、`Boolean_写入_true`。
- 标题修改时不修改编号。

### 用例类型

| 类型 | 含义 | 结果处理 |
|------|------|---------|
| 回归 | 产品行为已明确 | 按断言自动判定 Pass / Fail |
| 探索 | 产品规则或接口行为尚未确认 | 记录实际结果，暂不作为产品回归结论 |
| BLOCKED | 当前接口或工具能力不支持 | 不执行，保留占位 |

### Mock 节点选择

- 读取类用例使用 `*_r_*` 节点：`change=true`、`writable=false`，新增位号时 `onlyRead=true`。
- 写入类用例使用 `*_w_*` 节点：`change=false`、`writable=true`，新增位号时 `onlyRead=false`。
- 除专门测试异常映射外，`dataType` 必须与 UA 源端节点类型一致。

### 公共新增闭环

适用的新增成功用例至少验证：

1. `add_tag` 返回成功；
2. `tag-info/page` 按 `tagName` 查询到唯一记录，配置字段与请求一致；
3. `getRTValue` 在超时时间内返回有效值；
4. `queryWithQuality` 返回相同位号，值与 `getRTValue` 一致，质量码有效；
5. 关键映射、类型和写入用例使用 asyncua 直读源端作为独立对照；
6. 用例结束后删除新增位号；写入类用例恢复 UA 源端原值。

### 数据表示约束

- Int64 / UInt64 超过 JavaScript 安全整数范围时，测试工具必须以十进制字符串传输和比较，不得先转换为 JS `number`。
- Float / Double 使用误差比较，不使用字符串直接比较。
- DateTime 读取结果解析 `utcTime`，按 OPC UA 1601 epoch、100ns 单位转换为 UTC 后比较。

## 13 种数据类型：默认读取

每条读取用例均使用对应类型的 `*_r_*` 节点，并执行：新增位号、等待两个采集周期、两次读取 RT、调用 `queryWithQuality`、asyncua 直读源端。

公共读取断言：

1. 位号配置 `dataType` 与请求一致；
2. 两次 RT 值发生变化；
3. RT 值经类型归一化后与源端当前值一致；
4. `getRTValue` 与 `queryWithQuality` 返回值一致；
5. 质量码有效，时间戳可解析。

## 执行与清理要求

1. 每次运行生成唯一前缀，例如 `ua21_{runId}_`，避免与历史环境数据冲突；
2. 每条用例记录创建的数据源、位号 ID、系统位号名和底层节点名；
3. 写入前保存 UA 源端原值，结束后恢复并验证恢复成功；
4. 新增位号在用例结束后删除；若进入回收站，需要按测试环境约定物理清理；
5. 清理失败单独记录为 `cleanup_failed`，不得覆盖原测试结论；
6. 探索用例必须保存完整请求、响应、最终配置、RT、质量、源端值和历史证据，供后续确定产品规则并转成回归用例。
