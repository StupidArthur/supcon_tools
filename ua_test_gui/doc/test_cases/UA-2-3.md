# UA-2-3 位号导入导出

验证用户能否导出已有位号、使用 Excel 批量创建或更新位号，并从数据源选择底层节点批量注册位号。

本文件按用户任务组织，不按 API 拆分。API 仅用于构造数据、执行等价操作或独立核验。

---

## 设计约定

### 功能边界

覆盖：位号导出、Excel 导入、冲突处理、文件校验、导出后修改再导入、从数据源批量注册。

不覆盖历史值 JSON、Excel/ZIP 或 CSV 导入。

### 边界关系

- `UA-2-1`：单个位号新增及批量新增冒烟。
- `UA-2-2`：查找位号和浏览可注册节点。
- `UA-2-3`：完整导入导出和批量注册流程。
- `UA-6`：超大文件、最大数量和性能压力。

### 执行原则

1. GUI 有对应功能时优先按用户路径执行；API 用于独立核验。
2. 导入成功不能只看成功提示，必须回查数量、配置和可用性。
3. 导出结果按用户选择的稳定 ID 集合核对，不依赖页面顺序。
4. 从数据源批量注册前，使用 `dsId + tagBaseName` 排除已注册节点。
5. 临时数据统一使用 `ua23_{runId}_` 前缀并在结束后清理。

### 基础测试数据

- 两个正常数据源 A、B；存在跨数据源同名底层节点；
- 至少 13 个已注册位号，覆盖 13 种数据类型；
- 至少两个分组，配置单位、频率、只读、推送、描述、量程和报警限；
- A 中至少 20 个未注册底层节点；
- 至少一个无实时值或数据源断线的位号。

## 执行与清理要求

1. 每次运行生成唯一前缀 `ua23_{runId}_`；
2. 导入前保存目标名称、ID 和配置快照；
3. 导出文件使用独立临时目录并记录文件校验值；
4. 导入完成后按名称集合核对，不只检查接口成功码；
5. 成功导入的位号必须删除，覆盖的原位号必须恢复；
6. 清理失败记录为 `cleanup_failed`，不得覆盖原测试结论；
7. 探索用例保存文件、请求响应、配置前后快照和最终规则。
