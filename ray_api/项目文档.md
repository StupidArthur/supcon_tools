## 1. 项目定位

桌面 GUI 工具，监控多个互相独立的 Ray 集群（KubeRay 部署）。
- 采集 Ray 平台 REST 接口，存 SQLite 历史
- 展示节点/进程/Actor/作业的当前态与趋势
- CPU/内存/GPU 超限告警（4 状态机，需人工确认消除）
- 多集群统一管理，故障隔离

**技术栈（dev-skill 强制约束）**：Wails v2 + Go + React 18 + TypeScript + Shadcn UI + Tailwind v3 + SQLite(modernc 纯 Go 驱动)。GUI 固定 Wails+Go，无选型空间。

## 4. 数据流与采集

### 采集架构
```
CollectorManager (管N个)
  └─ Collector (每集群1个，独立goroutine)
       ├─ summary ticker (SampleEvery秒) → /nodes?view=summary → 节点硬件
       └─ detail ticker  (SampleEvery秒) → /nodes/{id}并发 + /api/cluster_status + /api/jobs/
```
- **统一采样间隔** `cfg.SampleEvery`（默认10秒），summary和detail共用（用户要求统一，不再分集群/分类型）
- **detail 节点详情限流并发**：信号量上限 `cfg.Concurrency`（默认10），全部完成后一次性更新快照
- **故障隔离**：每集群独立 goroutine + 独立 ctx；detail内 cluster/jobs 失败不影响节点详情
- **采集后回调 onAlert**：summary后查节点指标，detail后查worker指标

### 前端数据加载（关键性能点）
- 每5秒轮询：所有集群只拉**轻量 status**（状态点用）
- **只拉当前选中集群**的全量 snapshot + perf（避免N集群×大对象卡前端，这是之前卡顿的修复）
- 切换集群立即拉一次该集群快照
- 全局告警计数 `countAlerts('')` 每轮拉

### 数据来源对应（用户问过）
- **概览"分配值"**：`/api/cluster_status` 的 autoscalingStatus 文本，正则提取 CPU/内存/GPU 分配量
- **概览"实际占用"**：`/nodes?view=summary` 各节点 cpu/mem 汇总
- **进程名(Worker Process Name)**：`/nodes/{id}` workers[].coreWorkerStats[0].actorTitle，取第一个`(`前加`ray::`；IDLE→`ray::IDLE`
- **worker GPU**：按pid汇总该进程上actor的 requiredResources["GPU"]

## 5. 告警系统

### 状态机（恢复与消除分离，确认对整条报警）
```
未报警 ─超限─▶ 报警-未确认 ─确认─▶ 报警-已确认
                 │                    │
                 │ 连续N次低于         │ 连续N次低于
                 ▼                    ▼
           已恢复-未确认 ─确认─▶ 已消除(结束) ──再超限──▶ 新报警
```
- 两个布尔维度：recovered × acknowledged，都true才消除
- **触发**：单次超限即触发（无活报警则新建）
- **恢复**：连续 N 次（`RecoverConsecutive` 默认3）低于限值
- **确认**：用户操作，对整条报警全程有效；确认已恢复的报警立即消除
- 未消除期间反复超限/恢复记在同一报警下（alert_event表）
- 对象标识：node用nodeId，worker用`nodeId:pid`（**不用进程名**，避免同名串扰）

### 告警指标（受Ray数据限制）
- 节点 MEM/GPU：有占用率，能报 ✅
- 节点 CPU：缺核数算不出利用率，**暂不报** ⚠️
- worker CPU：有百分比，能报 ✅
- worker MEM/GPU：worker无总量，**暂不报** ⚠️

### 阈值
- 全局 `cfg.Thresholds`（CPU/MEM 80%，GPU 90%）
- 集群级覆盖：`ClusterConfig.Thresholds`（当前UI未暴露集群级，只全局；后端支持）
- 阈值=0 表示不告警

## 6. 配置

### config.json 结构（exe同级，写不进则回退 `%USERPROFILE%\ray_monitor\config.json`）
```go
Config {
  Clusters []ClusterConfig  // 每个只有 id + platformUrl（URL即集群名）
  SampleEvery int           // 统一采样间隔，默认10
  Thresholds Thresholds     // 全局报警阈值
  SortBy string             // cpu|gpu
  // 后端用，前端不暴露：TimeoutSec/Concurrency/GlobalConcurrency/RecoverConsecutive
}
ClusterConfig { ID, PlatformURL string }
```
- 旧格式（v1单platformUrl，或v2带name/cookie/集群级间隔）启动时自动迁移
- 集群显示名 = URL的 host:port（`ClusterConfig.DisplayName()`）
- 配置保存：`SaveConfig` → 集群列表变更走 SyncClusters 热增减；采样间隔变更走 ReloadAll 重建所有采集器

## 9. 给下一个会话的接手要点

1. **先读 `Ray监控工具设计v2.md`** 了解整体设计
2. **环境**：每条 bash 前 `export PATH="/d/TDM-GCC-64/bin:$PATH"`
3. **改 app.go 方法签名后必须 `wails generate module`**
4. **前端 Config/ClusterConfig 类型**：在 `api.ts` 里是纯 interface（剥离了 Wails 生成的 convertValues 方法），不要直接用 `config.Config` 类型做 setState，会缺 convertValues 报错
5. **用户工作风格**（重要，见 memory `feedback-evidence-before-assumption`）：
   - 排查问题**先找证据再下结论**，不要预设用户操作出错
   - 加诊断让程序自己报告状态（如 debug.txt、日志），不要猜
   - 每个假设要验证（写测试或加日志），不能跳过验证直接改代码
   - "需重启采集生效"这类标记必须真正实现重建逻辑，不能只标记
6. **当前优先**：确认告警显示问题是否已随 CreateAlert 修复解决。用户反馈后继续。
