## 设计理念

### 1.1 核心思想

本消息中间件的设计遵循以下核心理念：

#### **轻量级优先**
- 避免引入重量级的消息队列系统（如 RabbitMQ、Kafka）
- 利用项目已有的 Redis 基础设施
- 最小化协议开销，追求低延迟和高性能

#### **解耦与独立**
- 服务间通过消息总线通信，避免直接依赖
- 每个服务可以独立部署、独立扩展
- 支持服务的热插拔和动态发现

#### **简单易用**
- 提供简洁的 API，降低使用门槛
- 隐藏底层实现细节，开发者只需关注业务逻辑
- 支持多种通信模式，适应不同场景

#### **可靠性保证**
- 消息超时机制，避免无限等待
- 错误处理和重试机制
- 健康检查和心跳监控

### 1.2 为什么选择 Redis

#### **已有基础设施**
- 项目已经使用 Redis 进行实时数据推送
- 无需引入新的中间件，降低系统复杂度

#### **性能优势**
- Redis 是内存数据库，延迟极低（本地连接 < 1ms）
- 支持高效的 List、Hash、Pub/Sub 操作
- 单连接可处理 10,000+ 并发请求

#### **功能完备**
- **List**：实现命令队列（FIFO）
- **Hash**：存储响应结果（Key-Value）
- **Pub/Sub**：实现发布-订阅模式
- **String**：实现健康检查和状态存储

#### **轻量级**
- 相比 HTTP Server，无需维护 HTTP 协议栈
- 相比 RabbitMQ，无需额外的消息代理进程
- 相比 Kafka，无需复杂的集群配置

### 1.3 设计原则

#### **单一职责**
- `MessageBus`：核心消息路由和传递
- `MessageServer`：服务端消息处理
- `MessageClient`：客户端消息发送
- `ServiceRegistry`：服务注册与发现

#### **开闭原则**
- 消息格式可扩展（通过 `payload` 字段）
- 支持自定义消息类型（通过 `MessageType` 枚举）
- 处理器可动态注册和注销

#### **依赖倒置**
- 服务间通过消息总线通信，不直接依赖
- 使用接口抽象，便于替换实现

## 消息协议

### 3.1 消息格式

```python
{
    "message_id": "uuid-1234",           # 消息唯一ID
    "message_type": "request",           # 消息类型
    "service_name": "engine_service",    # 服务名称
    "action": "load_config",             # 操作类型
    "payload": {                         # 消息载荷
        "config_path": "/path/to/config.yaml",
        "namespace": ""
    },
    "request_id": null,                  # 请求ID（响应时使用）
    "timestamp": 1234567890.123,         # 时间戳
    "ttl": 60                            # 生存时间（秒）
}
```

### 3.2 消息类型

#### **REQUEST（请求）**
- 用于同步调用，需要等待响应
- `request_id` 为 `null`（由系统生成）
- 响应消息的 `request_id` 关联原请求的 `message_id`

#### **RESPONSE（响应）**
- 响应 REQUEST 消息
- `request_id` 关联原请求的 `message_id`
- `payload` 包含 `status` 和 `data` 或 `error`

#### **COMMAND（命令）**
- 单向命令，不需要响应
- 用于异步操作，如日志记录、状态更新

#### **EVENT（事件）**
- 发布-订阅模式
- 可以广播给多个订阅者
- 不需要响应

### 3.3 响应格式

#### **成功响应**
```python
{
    "status": "ok",
    "data": {
        "message": "Config loaded successfully"
    }
}
```

#### **错误响应**
```python
{
    "status": "error",
    "error": "Config file not found"
}
```

## 通信模式

### 4.1 命令-响应模式（Request-Response）

#### **适用场景**
- 需要立即获取结果的操作
- 配置加载、参数修改、状态查询等

#### **使用示例**

**服务端**：
```python
def handle_load_config(payload):
    config_path = payload["config_path"]
    engine.load_config(config_path)
    return {"message": "Config loaded"}

server.register_handler("load_config", handle_load_config)
```

**客户端**：
```python
result = client.call(
    "engine_service",
    "load_config",
    {"config_path": "/path/to/config.yaml"}
)
```

#### **特点**
- 同步阻塞，等待响应
- 自动超时处理
- 错误自动传播

### 4.2 异步命令模式（Async Command）

#### **适用场景**
- 不需要立即结果的长时间操作
- 日志记录、状态更新等

#### **使用示例**

**客户端**：
```python
# 发送异步命令
message_id = client.call_async(
    "engine_service",
    "export_data",
    {"output_path": "/path/to/output.csv"}
)

# 稍后获取结果
result = client.get_response(message_id, timeout=300)
```

#### **特点**
- 非阻塞，立即返回
- 可以稍后查询结果
- 适合长时间运行的操作

### 4.3 发布-订阅模式（Pub/Sub）

#### **适用场景**
- 事件通知
- 状态变更广播
- 多服务协同

#### **使用示例**

**发布者**：
```python
client.publish("config_updated", {
    "config_path": "/path/to/config.yaml",
    "timestamp": time.time()
})
```

**订阅者**：
```python
def on_config_updated(message):
    print(f"Config updated: {message.payload}")

bus.subscribe_events(
    ["config_updated"],
    on_config_updated
)
```

#### **特点**
- 一对多通信
- 解耦发布者和订阅者
- 支持事件过滤

### 4.4 服务发现模式（Service Discovery）

#### **适用场景**
- 动态服务注册
- 服务健康监控
- 负载均衡

#### **使用示例**

**服务注册**：
```python
registry.register("engine_service", {
    "version": "1.0.0",
    "endpoints": ["load_config", "patch_params"]
})
```

**服务发现**：
```python
# 发现服务
service_info = registry.discover("engine_service")

# 检查健康状态
is_healthy = registry.check_health("engine_service")

# 列出所有服务
services = registry.list_all()
```

#### **特点**
- 自动服务注册
- 健康状态监控
- 支持服务元数据

### 5.2 完整示例

#### **Engine Service 端**

```python
from components.message_bus import MessageBus, BusConfig, MessageServer
from controller.engine import UnifiedEngine

# 创建消息总线
bus = MessageBus(BusConfig(
    redis_host="localhost",
    redis_port=6379,
    key_prefix="data_factory"
))

# 创建引擎
engine = UnifiedEngine.from_program_config(config)

# 创建服务端
server = MessageServer("engine_service", bus)

# 注册处理器
def handle_load_config(payload):
    """加载组态"""
    config_path = payload["config_path"]
    namespace = payload.get("namespace", "")
    engine.load_config(config_path, namespace)
    return {"message": "Config loaded"}

def handle_patch_params(payload):
    """修改参数"""
    instance_name = payload["instance_name"]
    params = payload["params"]
    for key, value in params.items():
        engine.queue_param_update(instance_name, key, value)
    return {"message": "Params patched"}

def handle_get_snapshot(payload):
    """获取快照"""
    # 从 Redis 读取最新快照（因为 Engine 已经推送到 Redis）
    import redis
    redis_client = redis.Redis(decode_responses=True)
    snapshot_json = redis_client.get("data_factory:current")
    if snapshot_json:
        return json.loads(snapshot_json)
    return {}

server.register_handler("load_config", handle_load_config)
server.register_handler("patch_params", handle_patch_params)
server.register_handler("get_snapshot", handle_get_snapshot)

# 启动服务
server.start()

# 保持运行
import signal
signal.pause()
```

#### **WebServer 端**

```python
from components.message_bus import MessageBus, BusConfig, MessageClient
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

# 创建消息总线
bus = MessageBus(BusConfig(
    redis_host="localhost",
    redis_port=6379,
    key_prefix="data_factory"
))

# 创建客户端
client = MessageClient(bus, "webserver")

class ParamPatch(BaseModel):
    params: dict

@app.patch("/realtime/instances/{name}/params")
def patch_instance_params(name: str, req: ParamPatch):
    """修改实例参数"""
    try:
        result = client.call(
            "engine_service",
            "patch_params",
            {
                "instance_name": name,
                "params": req.params
            }
        )
        return {"status": "ok", **result}
    except TimeoutError:
        raise HTTPException(status_code=504, detail="Service timeout")
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

@app.get("/realtime/snapshot")
def get_snapshot():
    """获取快照"""
    try:
        snapshot = client.call("engine_service", "get_snapshot", {})
        return snapshot
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

## 总结

本消息中间件设计遵循**轻量级、解耦、简单易用**的原则，通过 Redis 实现高效的服务间通信。主要特点：

1. **性能优异**：低延迟、高并发、低资源消耗
2. **易于使用**：简洁的 API，清晰的文档
3. **功能完备**：支持多种通信模式和服务发现
4. **可扩展**：支持自定义扩展和优化

适用于微服务架构中的服务间通信，特别是对性能要求较高的场景。
