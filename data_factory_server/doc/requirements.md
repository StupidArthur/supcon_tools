# Data Factory Next 需求文档

## 1. 产品概述
Data Factory Next 是一个高性能的工业仿真与数据生成平台，旨在为上位机系统（如 SCADA、MES、ERP）提供高保真的实时数据流。它通过分布式的引擎架构，能够模拟大规模的测点信号，并支持历史数据回放。

## 2. 核心需求

### 2.1 多引擎并发运行
- **需求描述**：系统必须支持同时运行多个独立的计算引擎实例。
- **业务价值**：实现不同工艺逻辑的物理隔离，提高系统在大规模仿真下的伸缩性。
- **约束**：不同引擎之间的位号（Tag）命名空间通过引擎 ID 进行隔离。

### 2.2 异构计算逻辑支持
- **需求描述**：除了基于 DSL 的表达式计算外，系统需支持从 Excel/CSV 文件中回放历史数据。
- **业务价值**：允许用户基于真实的现场记录进行系统调试和算法验证。
- **约束**：回放数据在协议层（OPCUA）的表现形式必须与实时计算数据一致。

### 2.3 统一数据总线
- **需求描述**：所有引擎产生的数据必须汇总到统一的 Redis 数据总线上。
- **业务价值**：消除数据孤岛，使下游的存储和协议转发服务能够透明地处理所有数据。

### 2.4 自动化数据持久化
- **需求描述**：存储服务必须能够自动发现并收割所有活跃引擎的数据，将其持久化到 DuckDB 历史库中。
- **业务价值**：为后续的趋势分析、性能评估和报告生成提供原始数据支撑。

### 2.5 标准协议发布 (OPCUA)
- **需求描述**：系统需提供统一的 OPCUA Server，对外暴露所有引擎产生的测点。
- **业务价值**：通过标准工业协议与第三方系统无缝对接。

### 2.6 基础设施编排与热重载
- **需求描述**：系统需提供管理界面，允许用户通过 YAML 脚本编排引擎实例数量、类型及关联组态，并支持无需重启主程序的“全量热重载”。
- **业务价值**：实现生产环境下的动态扩容与逻辑调整，提高系统的在线可维护性。
- **约束**：重载过程需保证数据一致性，避免位号冲突。

## 3. 非功能性需求
- **高性能**：在单机环境下需支持万级位号的毫秒级推送。
- **可靠性**：单个引擎进程崩溃不应影响其他引擎及其关联服务的运行。
- **可维护性**：通过 Web 诊断页面直观展示所有服务的健康状态和执行性能。

## 4. Program 与数据模拟（2025-03）
- **CYLINDRICAL_TANK** 使用外部 `inlet_flow`，输出 `outlet_flow`，支持级联水箱。
- **VALVE** 根据 `inlet_flow` 与开度计算 `outlet_flow`。
- 每个 Program 必须定义 `param_descriptions`；默认展示/导出列**仅**由 DSL **非空** **`display_args`**（可选 `attr[ref]`）声明；未写或 `[]` 则该 program 项不参与；`ref` 仅用于预览图表纵坐标。导出在未指定 `selected_variables` 时与 `display_args` 列一致。数据模拟页支持单/双 Y 轴切换，无位号勾选对话框。
- API：`/simulate/preview` 返回 `display_variables`、`variable_meta`、`plot_scales`；`/export/run` 支持 `selected_variables`。
