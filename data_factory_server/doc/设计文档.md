# Data Factory Next 设计文档 (V2.0)

## 1. 架构架构 (System Architecture)

系统采用 **“生产者-总线-消费者” (Producer-Bus-Consumer)** 的流水线架构，依托 Redis 实现各组件的完全解耦。

### 1.1 核心组件
- **ConfigServer (组态中心)**：加载 `engines_manifest.yaml`，协调所有服务。
- **UnifiedEngine (仿真引擎)**：执行 DSL 算法，计算实时数据。
- **PlaybackEngine (回放引擎)**：加载静态数据文件，按时钟回放。
- **RealtimePublisher (发布器)**：将数据写入 Redis，支持 `engine_id` 路由。
- **StorageService (存储服务)**：订阅总线数据，写入 DuckDB。
- **OPCUAServer (协议网关)**：提供标准 OPC UA 访问接口。
- **ServiceManager (服务管理器)**：管理所有独立进程的生命周期。

## 2. 详细设计

### 2.1 数据总线模式 (Redis V2)
- **实时快照**：存储在 Redis Hash `data_factory:v2:current`。
  - **Key**: `tag_name` (例如 `ns2.tank_01.pressure`)
  - **Value**: JSON 字符串 `{"value": 12.3, "timestamp": 1700000000, "engine_id": "sim_1"}`
- **兼容策略**：运行时已移除 Redis V1 `data_factory:current` 的读写路径，统一走 V2。
- **全局注册表**：`data_factory:registry:tags` (Hash)，记录每个位号所属的引擎和类型。

### 2.2 组态分发机制
1. `ConfigServer` 启动，扫描并解析所有 DSL 文件。
2. 自动注入 `engine_id` 作为命名空间前缀。
3. 执行位号唯一性检查，若冲突则终止启动。
4. 将合并后的组态和采样策略写入 Redis。
5. 发布 `config_published` 事件。
6. 各下游服务监听事件并重载 (Hot Reload)。

### 2.3 执行引擎 (Engine & Playback)
- **BaseRuntime 接口**：定义了 `step()`, `get_statistics()`, `get_metadata()` 等标准方法。
- **JIT 加速 (TaskRuntime)**：当 `sleep_time=0` 时，引擎进入紧凑执行模式，仅受限于 CPU 算力。
- **性能优化**：
  - 使用 **NumPy** 预生成随机池。
  - 使用 **Python Abstract Nodes** 树结构，减少表达式解析开销。

### 2.4 存储策略
- `StorageService` 不再绑定特定引擎，而是根据 `ConfigServer` 下发的“存储清单”进行全局收割。
- 数据库表结构扩展了 `engine_id` 和 `source_logic` 字段，支持混合查询。

## 3. 接口设计 (API)

### 3.1 诊断与管理 API
- `GET /api/v2/services/status`: 获取所有服务的实时健康度。
- `GET /api/v2/engines/{id}/stats`: 获取特定引擎的计算压力和频率。
- `POST /api/v2/system/reload`: 触发生态系统的热重载。

## 4. 前端设计

- **组件化诊断**：
  - `TopologyMap`: 基于 ECharts 的关系图，动态展示引擎与消费者的连接关系（心跳驱动）。
  - `EngineSelect`: 一个持久化的筛选组件，允许用户在控制台切换正在查看的引擎。

## 5. 关键算法

### 5.1 随机性优化 (NumPy Pool)
为了避免高频调用 `random.uniform` 产生的 GIL 竞争和计算浪费，引入预生成的 NumPy Pool。引擎每次 `execute` 从 Pool 中顺序读取，极大提升吞吐量。

### 5.2 命名空间重写 (AST Rewriter)
通过 Python `ast` 模块，递归遍历 DSL 中的表达式。例如：
`expression: "tank1.level * 0.5"` -> 在引擎 `sim0` 中重写为 `sim0.tank1.level * 0.5`。
这保证了用户在编写 DSL 时无需关心部署在哪个引擎。

### 5.3 Program 展示与 DSL `display_args`
- **BaseProgram** 约定子类必须实现：
  - `stored_attributes`：进入 VariableStore/快照/历史的全部对外标量。
  - `param_descriptions`：各 stored 键的中文说明，供位号配置表展示。
- **默认展示列与绘图缩放**：由 DSL 每条 `program` 的可选字段 `display_args` 声明（`attr` 或 `attr[ref]`）；解析为 `display_specs` 存入 `ProgramItem`。`ref` 仅用于数据模拟页 ECharts 纵坐标缩放：`y_plot = y_raw * (100/ref)`；**运行态与 CSV 导出始终为 `y_raw`**。
- **UnifiedEngine** 提供 `get_variable_meta()`（含 `is_display`、`plot_scale_ref`）、`get_display_variables()`、`get_plot_scales()`；逻辑完全基于 `display_specs` 与 VARIABLE 规则，不再读取类级 `display_attributes`。
- **CYLINDRICAL_TANK / VALVE**：入口/出口流量在模型边界上显式传递，便于级联；详见各模块 `doc` 与 `params_table`。

### 5.3.1 PID 运行模式（MODE）
- **PID** 在 `stored_attributes` / `default_params` / `param_descriptions` / `params_table` 中增加 **`MODE`**（默认 `1`）。`execute` 内仅当数值 **`MODE == 1`** 时执行比例/积分/微分与 `MV` 更新；否则本周期直接返回（不改动 `MV`、`_integral`、`_last_error`）。可选形参 `execute(..., MODE=...)` 用于按周期覆盖实例属性。

### 5.4 TAG 程序（统一可写位号）
- 新增 `TAG` 算法程序，`stored_attributes = ["in_value", "out"]`，用于“可写输入 + 可读输出”的统一位号建模。
- `TAG.execute(in_value=None)`：
  - 未传 `in_value` 时：保持当前 `in_value`，执行 `out = in_value`；
  - 传入 `in_value` 时：先更新 `in_value`，再执行 `out = in_value`。
- 设计约束：输入位号统一命名为 `in_value`，避免关键字冲突，表达式层与写值接口保持同名。

### 5.5 导出列与多格式导出
- `CSVExporter.export(..., column_keys=...)` 与 `UnifiedEngine.export_to_csv(..., selected_variables=...)`：未指定 `selected_variables` 时由 `run_export` 填入 `get_display_variables()`（DSL 非空 `display_args`），仅导出这些数据列。
- `ExportTemplate` 采用前端同构字段：`header_rows`、`title_names`、`time_format`、`file_format`、`sheet_name`；并通过属性派生 `time_column_name` 与 `time_row2_description` 供 `CSVExporter/ExcelExporter` 复用。
- `UnifiedEngine.export_snapshots(...)` 按内存中的 `ExportTemplate` 与 `file_format` 调用 `CSVExporter` 或 `ExcelExporter`（`xlsx`/`xls`）。
- `run_export(..., export_format=...)`：提供时构建内联 `ExportTemplate`，不再从 YAML 读取导出样式；`services/export_runner.py` 统一修正输出扩展名。
- `TemplateManager._parse_template(...)` 不再兼容旧字段（`time_column_name` 等），仅接受 `defaults` 对象，实现模板能力单一来源。

### 5.6 模板加载链路（2026-03）
- 模板目录：`components/export_templates/templates/*.yaml`。
- 列表接口：`GET /templates/list` 由 `TemplateManager.list_templates()` 动态扫描文件名。
- 默认值接口：`GET /export/format-defaults/{template_name}` 直接返回 `ExportTemplate.to_export_format_defaults()`。
- 前端 `DataSimulation` 在打开导出弹窗与切换模板时调用默认值接口，右侧导出参数表单始终由模板数据初始化。

### 5.7 独立合成数据文件生成（开发/测试）
- 模块：`components/export_templates/synthetic_export_file_generator.py`。
- 入口函数：`generate_synthetic_export_file(tags, template, starttime, endtime, interval=5.0)`。
- 行为：按 `starttime`～`endtime` 与 `interval` 生成时间序列；各列数值为 **100～199** 整数锯齿波（与模板 `time_format` 无关）；输出格式由模板 YAML 的 `file_format`（`csv` / `xlsx` / `xls`）决定，表头与 `CSVExporter` / `ExcelExporter` 一致。
- 默认输出路径：`components/export_templates/generated_data/{template}_synthetic_{时间戳}.{扩展名}`（目录已加入 `.gitignore`）。
